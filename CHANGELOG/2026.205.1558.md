## What changed

- **W4 pure-helpers relocation** — `ClassifyNodeTarget`/`PathLeaf` promoted
  to `sdk/deploykit` (pure functions of `(node, path)`, no `LoadUnified`/
  executor dependency), shared by charly-core's
  `deriveChildExecutorForPath` and `candy/plugin-bundle`'s walk
  classification (R3). `#DeployNodeDispatchRequest` gains `target`/
  `vm_entity` fields so the plugin's pre-resolved classification crosses
  the wire instead of the host recomputing it. `#DeployEntityResolveRequest.kind`'s
  documented vocabulary extended with `"local"` (doc-comment only).
- **K5-A item 1 — compile-seam ctx-threading.** Retires the core-`init()`-set
  `deploykit.CompileActOp`/`CompileServiceSteps` func-var indirection (a
  hidden, implicit same-process dependency — silently nil for any caller
  that didn't share it). `BuildDeployPlan`/`CompileOpSteps` now take
  explicit `ctx`/`exec` parameters. The common case (every non-`plugin:`-verb
  op; the packaged/supervisord service cases) is fully portable and never
  touches the wire. Two genuinely host-only pieces reach back over new
  CUE-sourced HostBuild seams: `#ConstructStepRequest`/`#ConstructStepReply`
  (a `run: plugin: <word>` op resolving against the provider registry) and
  `#RenderServiceRequest`/`#RenderServiceReply` (rendering a systemd CUSTOM
  unit, which needs `candy/plugin-init`'s `OpResolve` + the M16 egress
  gate). A dead lazy-fallback path (`LoadBuildConfigForBox`'s per-call
  compile outside the deploy-compile seam) is deleted, not ported — it was
  dead in production and every test exercising it relied on it FAILING.
- **K5-A item 2 — `DescribeProvider` RPC + pod-overlay dispatch
  relocation.** New `ExecutorService` RPC, CUE-sourced
  (`sdk/protocol/schema/plugin.cue` → `task wire:gen`):
  `DescribeProvider(class, word)` → cached capability metadata (today:
  `#StepContract`) for another provider — the metadata twin of
  `InvokeProvider` (dispatch) and `HostBuild` (host-side build). Lets a
  plugin make a routing decision about a peer provider (e.g. "does
  `class:step` word X declare `Emits=true`?") without holding the provider
  registry itself or paying a live `Invoke` round-trip merely to query
  shape. `#DescribeProviderRequest{class, word}` /
  `#DescribeProviderReply{found, step_contract}` — `found=false` is a QUERY
  RESULT, not an RPC error (mirrors `InvokeProvider`'s "not connected"
  outcome). Client: `sdk.Executor.DescribeProvider(ctx, class, word) ->
  (found bool, *StepContract, error)`. The kind→word vocabulary for the 12
  compiler-emitted `InstallStep` kinds (`pluginEmitStepWords`, formerly
  charly-private) relocates to `sdk/deploykit.PluginEmitStepWords` — a
  genuine cross-boundary shared constant now that the pod-overlay
  step-emit DISPATCH DECISION itself moves plugin-side (`charly`-side
  companion: `candy/plugin-installstep`'s new `oci-dispatch` word).
- **F11 impact** — `DescribeProvider` is a NEW method on
  `ExecutorServiceClient`; `charly/uniform_api_gate_test.go`'s fixed
  method-set assertions were updated in the consuming (charly) PR, and the
  RPC itself is class/word-generic (never names a plugin word in the
  method signature — F11-clean by construction).
