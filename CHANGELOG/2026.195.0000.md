# 2026.195.0000 — VM extra-port-forward auto-allocation: `auto:<guest>` sentinel (sdk leg)

The sdk half of the VM extra-port-forward auto-allocation feature (superproject #65 — the R2
immediate-next from the live `16443` host-port collision between check-k3s-vm and
check-k8s-deploy-cluster, which share the `k3s-vm` template's hardcoded
`port_forwards: ["16443:6443"]`). Extends `ssh.port_auto` to the extra forwards: the host side of a
`network.port_forwards` entry may now be the `auto` sentinel, auto-allocated to a free host port at
vm-create and persisted for reuse. The superproject leg (plugin-vm allocation + orchestrator persist,
resolved `deployVMForwards`, the bed conversion) consumes this over the sdk pointer bump.

## Changed

- **`schema/vm.cue`** — `#VmNetwork.port_forwards` host side widened from the bare `=~":"` to
  `=~"^(auto|[0-9]{1,5}):[0-9]{1,5}$"`: each entry is `"<host>:<guest>"` where `<host>` is a fixed port
  OR the literal `auto` sentinel (matching the pod `port: [auto]` word). NOT a wire-key change and NOT
  a schema-version bump — `port_forwards` is unchanged, `auto` is purely additive, and the tightening
  only rejects strings that were always semantically invalid (the renderers already
  split-and-drop them). The generated `VmNetwork.PortForwards` stays `[]string` (a regex on a string
  element does not change the Go type — cue-exp-gengotypes-verified).
- **`schema/deploy.cue`** — `#VmDeployState` (the `@go(-)` open-tail machine-written state record) gains
  `port_forwards?: {[string]: int}` (guest-port → auto-allocated host port), the persisted sibling of
  `ssh_port` that makes the allocation stable across the vm-create → deploy-add sequence. Validation-only
  (the Go type is hand-mirrored).
- **`spec/hand_state_types.go`** — `VmDeployState` gains the matching hand field
  `PortForwards map[string]int` (kept in lockstep with the CUE def, as documented at the type).
- **`vmshared/qemu_render.go`** — `renderQemuNic` no longer reads `spec.Network.PortForwards` directly; it
  renders the extra `hostfwd` entries ONLY from the RESOLVED `rt.ExtraPortForwards` (the host-side-allocated
  `"host:guest"` strings the orchestrator produces), exactly as it already reads the resolved `rt.SshPort`
  for the `:22` forward — the spec=intent / rt=resolved rule. Regression-tested
  (`vmshared/qemu_render_test.go`): the renderer must NOT leak the `auto` sentinel or a raw spec host port.
