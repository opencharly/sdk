# 2026.195.0544 — K3-D+ — resolved-project envelope growth for the validate ENGINE relocation

The sdk contract leg of superproject task #60 (K3-D+, Units B+C): the `charly box validate` ENGINE moves
out of `charly/` core into the compiled-in `candy/plugin-box` plugin, which reads the resolved-project
**envelope** (`spec.ResolvedProject`) instead of the runtime `*Candy`/`*Config` graph. This grows the
envelope with exactly the fields the relocated rule engine consumes — no behaviour change in sdk itself;
the consumers land in the superproject.

## `#ResolvedProject` / `#CandyView` growth (schema/resolvedproject.cue)

- `#CandyCapabilitiesView` (NEW) + `capabilities?` on `#CandyView` — the per-candy capabilities aggregate
  the engine reads (orchestrator ruling a: the capabilities aggregate rides the envelope, not a re-derive).
- `plan?` + `authored_aliases?` on `#ResolvedBoxView` — the box-level plan + authored alias set the
  moved dup-alias / plan rules read off the box view.
- `provider_capabilities?` + `act_capable_verbs?` on `#ResolvedProject` (D-data word-sets:
  `provider_capabilities` = the `<class>:<word>` set; `act_capable_verbs` = the plugin words that act in
  build/deploy) — the kind-blind DATA the moved `validatePluginCandy` / `opActsInBuildDeploy` rules consult
  by word.
- `plugin_providers?` + `plugin_source?` on `#CandyView` — the validatePluginCandy subject (a plugin
  candy's declared providers + source).

## `#ValidateProjectReply` (schema/validateproject.cue, NEW)

The `validate-project` host-builder reply: `{Project (a TOLERANT partial `#ResolvedProject` — filled even
when LoadConfig/Scan/LoadUnified/ResolveBox errored, so validate runs on broken projects), Diagnostics}`.
The host-natural checks (CUE-conformance + build-tunable/merge) run host-side and their findings ride the
`Diagnostics`; the plugin engine merges them with its own per-kind/op findings into the verdict.

carries its pure label-set methods.
