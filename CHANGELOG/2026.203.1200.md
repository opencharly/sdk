# 2026.203.1200 — FLOOR-SLIM-proper Unit-8 sdk-contract leg (arbiter-bracket K4-exit + builder pre-pass wire)

The sdk half of Unit-8 of the core-minimization program: the wire contract
that lets `candy/plugin-bundle` own the Q1 resource-arbiter bracket call and
the deploy-time builder pre-pass, both moving out of `charly/` core.

## New / changed

- **`#DeployTargetDispatchRequest` gains `has_plan?: bool` (`@go(HasPlan)`)**
  (`schema/seam.cue`) — core's `deriveChildExecutorForPath`/`Start`/`Stop`
  threads whether this substrate registers a Start/Stop plan hook (today
  only "pod") onto the wire instead of bracketing the dispatch call itself.
- **`#ArbiterBracketAcquireRequest`/`Reply` + `#ArbiterBracketReleaseRequest`/`Reply`**
  (new CUE defs, `schema/seam.cue`) — the two HostBuild request/reply
  envelopes for the "arbiter-bracket-acquire"/"arbiter-bracket-release"
  seams charly-core registers (`host_build_arbiter_bracket.go`) and
  `candy/plugin-bundle`'s `handleLifecycleSimple` calls via
  `exec.HostBuild`, so the CHARLY_PREEMPT_LEASE `os.Setenv`/`os.Getenv`
  mutation still executes host-side (preserving the nested-`charly`-
  subprocess env-inheritance property) while OWNERSHIP of the bracket call
  — deciding when to acquire/release, around which dispatch — moves
  plugin-side.
- **`deploykit.BuilderPreresolved`'s doc comment updated** (`deploykit/compiler_deps.go`)
  to reflect that the Candy-coupled functions that build it now live in
  `candy/plugin-bundle` (over `exec.InvokeProvider`), not charly-core — no
  type or field change, comment-only.

## Why

Continues the FLOOR-SLIM-proper core-minimization program (see the
superproject's Unit-8 cutover): the arbiter-bracket move resolves a prior
round's overturned "stays core" permanence claim (the `arbiter_bracket.go`
UNTIL-K4 exit), and the builder pre-pass move is spike-proven live
(`InvokeProvider` dispatches `class=builder` to a real out-of-process
builder plugin end-to-end).
