## P12a — `kit/runner.go` (new): the check ENGINE relocates into `sdk/kit`
The check-engine **Runner** — the host context the plan walk drives (`RunOne`/`RunPlan` consume it) —
relocates from charly core into `sdk/kit` as an importable, CONSTRUCTOR-SHAPED library type, so any plugin
candy that runs a plan drives the SAME loop the in-core check engine uses. Its state is unexported and its
interface methods (`Mode`/`Distros`/`VerifyOnly`/…) would collide with same-named fields, so the host builds
it via `kit.NewRunner(kit.RunnerConfig{…})` and reaches its state through methods (`var _ PlanContext =
(*Runner)(nil)` locks the contract). Three injected host seams keep the host-coupled surfaces in core:
`PlanGrammar` (the do-mode/context grammar — `VerbCatalog`), `VenueResolver` (the `on:` per-step venue swap),
and the verb dispatch. The 12-method `kit.CheckContext` (its 6 host-reverse legs — `HTTPDo` + the four
`Resolve*` podman/go-libvirt/ssh ops) stays a thin core wrapper around `*Runner`. Behaviour-neutral:
`charly check box` is BYTE-IDENTICAL across every `--format` before vs after (the relocation's own parity gate).

## P12b — `sdk/schema/seam.cue`: the transitional check-family host seams (CUE-first; `task cue:gen`)
- **`#CheckRunRequest`** grown to the discriminated per-mode envelope (`mode!` ∈
  box|live|feature-box|feature-live|score|preflight, + name/image/instance/dir/section/filter/tag/strict/agent/
  timeout/no_agent/plan) — the `hostBuildCheckRun` atom the plugin's CLI leaves drive.
- **`#CheckBedRequest` / `#CheckBedReply` / `#CheckBedPrereqSkip` / `#CheckBedMember`** — the op-discriminated
  `check-bed` host-SESSION seam (setup/members-up/members-down/wait-ready/teardown): the lock/lease/env
  lifecycle the compiled-in bed runner cannot hold across a process boundary. The reply is the BedDescriptor
  the kind-blind plugin drives the R10 sequence from — **`BedDomain`** (per-deploy VM domain, #33: the plugin
  drives `charly vm create/destroy/start --domain <this>`, `vm build` entity-scoped via `VMTemplate`),
  **`Members` (`#CheckBedMember{key,is_vm,image,from}`)** (a group bed's per-member build coordinates for the
  member build + rebuild loop), **`LocalChildKeys`** (the host-rooted subset a VM root deploys host-side).
- **`#CheckConfigRequest` / `#CheckConfigReply`** — the dedicated `check-config` projection seam a plugin
  cannot LoadUnified for. Both check seams are TRANSITIONAL (K1/K5-mortal — die into the
  `#LoadedProject`/`#ResolvedProject` envelope spine). Class-generic action nouns only (F11).

## P12b — the wire types + the exit-code contract
- **`sdk/spec/deploy_wire.go` `CliRequest.Combined`** — merges stderr into the captured `CliReply.Stdout`
  (with `Capture`): a check-bed step's child `charly check …` writes its results to STDERR, so the plugin's
  per-step `.log` needs the combined stream (pre-relocation `runCapture` parity). One `*bytes.Buffer`, host-serialized.
- **`sdk/kit/checkrun_seam.go` `CheckRunReply` grown** — `Score *spec.CheckRunResults` (the harness scorer's
  "score" mode), `Header` (the kind-specific banner the host builds so the plugin stays kind-blind),
  `Passthrough *StepPass{Stdout,Stderr,ExitCode}` (nested-pod-in-VM guest delegation).
- **`sdk/spec` `CheckRunResults` / `StepScore` / `ScoreSummary`** — CUE-sourced (SDD), imported by BOTH the
  core `RunCheckLive` (score-mode body) AND the plugin scorer — no alias, one source.
- **`sdk/exitcode.go` (NEW) — the cross-module exit-code contract.** `ExitCodeError{Code,Err}` carries a
  command plugin's PROCESS exit code across the module boundary (the host cannot classify the plugin's own
  error types); `main()` honors it via `errors.As`. `CheckFailExitCode` (2) / `CheckSkippedExitCode` (3) — the
  check 0/1/2/3 convention, single-sourced here so both the host (`main()` + `charly box feature run`) and
  `candy/plugin-check` reference ONE contract.
- **`sdk/kit/runner.go` VmName doc** — describes the caller-set per-deploy DOMAIN IDENTITY (not the shared
  kind:vm entity), matching the #33 semantics the check runner threads.
