# 2026.192.0700 — the unified-config parse lands in sdk/loaderkit (P6)

The node-form config PARSE — yaml → the generic, sdk-expressible `spec.ParsedProject`
(reserved-word node decomposition + plan-step desugar) — moves from charly core into a new
`sdk/loaderkit` package, so the loader plugin candy (and, transitionally, charly core) parse a
project with the SAME code. This is the parse half of the P6 loader-to-plugin cutover; the
superproject consumer wires the loader plugin as the config front-end and deletes the in-core
parse.

- **`loaderkit.ParseDoc`** — decomposes one node-form document into `spec.ParsedProject`
  (`ParsedNode`s: name + kind discriminator + opaque JSON body + member children). Consults ONLY
  spec vocabulary + yaml + the host-threaded `loaderkit.Threaded` (registry-derived
  kind-recognition DATA — which words are recognized kinds / deploy substrates / member-nesting
  structural kinds, and each verb's scalar-sugar primary field), never a provider registry — a
  kind-blind mechanism (boundary law clause D).
- **`loaderkit.DocParser`** — the swappable per-document parse seam: the host resolves the
  registered loader provider to it and calls it for every document, so an alternative loader
  plugin serves a different config front-end. `loaderkit.DefaultParser` is the built-in.
- **`spec.ParsedProject` / `ParsedNode`** — the CUE-sourced (`schema/parsedproject.cue`) wire
  type the parse produces and the host materializes into the typed `*UnifiedFile`.
- **`schema/candy.cue`** — the `#PluginCapability` regex admits the new `loader:` provider class.

Standalone tests (`loaderkit/parse_test.go`) cover entity decode, directive skip, plan-sugar
desugar, member-child recursion, and the no-discriminator / duplicate-name errors.
