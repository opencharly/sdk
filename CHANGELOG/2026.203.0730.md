# 2026.203.0730 — bed-robustness batch: guest SSH hardening + VM state-dir scoping

The sdk half of the bed-robustness thematic batch (the accumulated non-blocking
VM/deploy/bed-reliability fixes surfaced across recent wave landings): the two
items whose fix lives in `sdk/vmshared` — guest cloud-init SSH hardening and a
worktree-scoping override for the VM host-state root directory.

## New / changed

- **Guest SSH hardening (D18) — `vmshared/cloud_init_render.go`.** Closes the
  RCA'd kex-reset wedge class: OpenSSH ≥ 9.8 defaults `PerSourcePenalties` ON,
  which penalizes repeated connection attempts from ONE source — exactly what
  `kit.WaitForSSH`'s readiness poll does, and every VM guest is reached through
  the SAME single passt gateway source IP, so the poll can trip its own
  guest's rate-limit and appear to "reset forever" against an otherwise-healthy
  guest.
  - `composeBootCmd` masks `ssh.socket` as the FIRST `bootcmd` entry (the
    earliest cloud-init phase) so a socket-activated sshd (enabled by default
    on some cloud images, notably Debian/Ubuntu) can never accept a connection
    before cloud-init has finished configuring the guest.
  - `composeRunCmd` writes and self-validates a `PerSourcePenalties no`
    sshd_config.d drop-in (`sshHardeningDropInCmd`) via a shell snippet — NOT
    a static cloud-init `write_files` entry — because the directive does not
    exist before OpenSSH 9.8: the snippet runs `sshd -t` after the write and
    deletes the drop-in again on failure, so an OLDER guest OpenSSH that
    rejects the unknown directive is never left with a config sshd refuses to
    start against (fail-safe to the pre-fix behavior, never a bricked sshd).
    `ssh.socket` is then unmasked and the resolved sshd unit enabled — the
    LAST charly runcmd step before the user's own — so sshd's first-ever
    start already reads a validated config, after cloud-init's own
    package/user/write_files work has completed.
  - **Design trade-off, documented deliberately (not overlooked):** masking
    `ssh.socket` until runcmd makes the guest **deterministically unreachable
    via SSH for the entire package-install phase**, trading the old
    "sometimes-flaky-but-reachable" window (sshd up early, racing a possible
    host-key rewrite) for "safe-but-fully-blocked-if-package-install-stalls."
    Verified live on `check-charly-vm` (a fresh VM build + an `update`
    fresh-rebuild — a second full pass through the sequencing) and
    `check-sidecar-pod`'s nested ephemeral VM (a fresh deploy + a full
    rebuild + redeploy cycle): all four passes completed in 1–3 minutes with
    zero wedge, now that the companion redundant-package-reinstall fix
    (`pacman -S --needed`, landed in the same wave) keeps the package-install
    phase itself fast. If field evidence ever shows a package-install stall
    under this ordering, the revisit path is either (a) moving the unmask
    earlier (e.g. a `write_files`-stage cloud-init module instead of
    `runcmd`), or (b) dropping the mask/unmask pair entirely and relying on
    the `PerSourcePenalties` drop-in alone.
- **`vmshared.VmStateRoot()` + `VmStateDirEnv`** (`vmshared/vm_state_root.go`,
  new) — the single root-path resolver for per-VM host state
  (`~/.local/share/charly/vm/<domain>/` — ssh keys, known_hosts, seed ISO, the
  per-domain disk overlay, `snapshots/`, `instance.yml`). Closes the "global
  `~/.local/share/charly/vm` non-worktree-scoping footgun": a libvirt DOMAIN
  name is derived purely from the deploy name, with no project/worktree
  component, so two concurrent `git worktree` checkouts of this repo running
  the SAME bed/deploy name (the common case — bed names like
  `check-charly-vm` are fixed identifiers shared across every worktree)
  collided on the same host state directory and the same libvirt domain.
  `VmStateRoot()` honors `CHARLY_VM_STATE_DIR` (must be an absolute path) as
  an override; every VM state-path call site in this package
  (`vm_snapshot.go`'s `snapshotsDir`/`vmDiskPath`) and in the superproject
  (`charly/vm_backend_lifecycle.go`, `charly/host_build_vm_build.go`,
  `candy/plugin-vm/vm.go`, `vm_create_orchestrate.go`,
  `vm_instance_override.go`, `candy/plugin-deploy-vm/lifecycle.go`) now routes
  through it instead of duplicating the literal path (R3). Default behavior
  (no env var set) is byte-identical to before this fix — purely additive.
