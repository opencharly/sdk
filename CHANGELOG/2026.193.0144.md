# 2026.193.0144 — P10 command-plugin config/build seam wire types (sdk producer)

The sdk producer half of P10 (VM CLI family → candy/plugin-vm). A COMPILED-IN command plugin owns its
CLI handlers but cannot reach core Mechanisms (the config loader, the deploy ledger, the VM-disk build
engine) — a plugin imports only the sdk. P10 adds the GENERIC host↔plugin seam wire types the plugin
exchanges with the host over the F10 reverse channel; the first consumer is command:vm, and the pod (P11)
+ bundle (P13) command families reuse the SAME class-generic seams.

- **spec/config_resolve_wire.go** — the "config-resolve" READ seam + "config-persist" WRITE seam:
  - `ConfigResolveRequest{Entity, Dir}` → `ConfigResolveReply{VM, Resources, Backend, Claimant,
    ClaimantNode, VmBackend, BuildEngine, RunEngine, VmState, VmEntities}` — the host resolves the
    project config (LoadUnified / ResolveRuntime / resolveVmBackend + #Vm defaults + the persisted
    ledger state) ONCE and ships it back; the plugin owns every downstream action.
  - `ConfigPersistRequest{Key, Entity, VmState, Remove}` → `ConfigPersistReply{}` — the WRITE twin; the
    host applies the deploy-ledger persist/remove under its blocking config lock (a core Mechanism the
    plugin cannot hold across the module boundary — a process-shared flock stays host-side).
  Hand-written wire (the `LifecyclePrepareInput` sibling pattern): the reply embeds the hand-written
  runtime types `spec.ResolvedVm`/`ResolvedResource`/`Deploy`/`VmDeployState`, which `cue exp gengotypes`
  cannot express without a colliding duplicate — the SDD "generator cannot express the shape" exception.
- **spec/vm_build_wire.go** — the "vm-build" seam: `VmBuildRequest{Box, Size, RootSize, Tag, Type,
  Transport, Console}` → `VmBuildReply{}`. The VM-disk build engine (privileged pacstrap / bootc /
  cloud-image → qcow2) STAYS core, exactly as the box-build engine stayed core behind HostBuild in P8;
  the plugin's thin `vm build` command forwards its flags and the host runs the whole build in-process.

The action nouns ("config-resolve" / "config-persist" / "vm-build") are class-generic — never a substrate
word — so the F11 uniform-API gate accepts them on the host-builder surface. Additive: no existing type
changed; no `#SchemaVersion` bump.
