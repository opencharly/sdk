# 2026.198.0002 ‚Äî K5: narrow status-substrate wire seam, drop host-side nested resolution

Schema half of the K5 "status remainder" lane: drops two fields from the
`command:status` plugin's host-collection wire contract now that the
declared nested-deployment-tree pre-resolution moves plugin-side (the
companion `charly` PR moves `charly/status_nested.go`'s logic into
`candy/plugin-status/nested_tree.go`, computing the tree directly off the
resolved-project envelope + `deploykit.LoadBundleConfig` ‚Äî both already
sdk-portable per the earlier android status-collector precedent, PR
#135/#136 in the `charly` repo).

## Change

`schema/status.cue`:

- `#StatusSubstrateRequest.nested` ‚Äî DROPPED. The host no longer needs to
  know whether `--nested` was requested; the plugin resolves the tree
  itself and decides independently whether to live-probe it.
- `#StatusSubstrateReply.roots` ‚Äî DROPPED. The host no longer pre-resolves
  and returns the declared nested roots; the plugin computes its own via
  the relocated `buildStatusRootsTree`.
- `#StatusNestedNode` ‚Äî KEPT (re-documented only). It stays a CUE def
  because it is self-recursive and the plugin still needs the generated Go
  type for its own internal tree-building, even though the type no longer
  crosses the host‚Üîplugin wire as part of `#StatusSubstrateReply`.

Regenerated `spec/cue_types_gen.go` fresh from `origin/main` + this schema
change only ‚Äî confirmed the diff carries zero interleaving from the other
two currently-open sdk PRs (#62, a `schema/seam.cue` check-bed-seam
narrowing; #64, a `deploykit`/`kit` shared-resolver extraction also
touching `schema/seam.cue`) ‚Äî neither touches `schema/status.cue`, so
there is no schema-file overlap with either.

## Change class + landing order

`charly`-or-sdk Go ‚Äî sdk half of a cross-repo cutover. **This half is safe
to land standalone**: the `charly` repo's `main` currently pins an OLDER
sdk commit, so nothing yet references the narrowed wire shape. The risk
this narrowing creates is entirely on the CONSUMER side ‚Äî bumping
`charly`'s sdk gitlink to this commit WITHOUT its consumer fix landing in
the same change would break the build (`charly/status_substrate_host.go`
still reads the now-dropped `req.Nested` + constructs
`spec.StatusSubstrateReply{..., Roots: roots}`; `candy/plugin-status/
command.go` + `provider.go` still read `reply.Roots` and set
`Nested: cmd.Nested` in the request literal) ‚Äî empirically confirmed via
an assembled-superproject build with this commit's gitlink staged against
the current `charly` main, which fails exactly as predicted. The companion
`charly` PR (K5 status-remainder: nested-tree pre-resolution ‚Üí
`candy/plugin-status`) fixes all three call sites in the same change and
bumps its `sdk` gitlink to this commit ‚Äî landing order: this PR (sdk)
merges first, then the companion `charly` PR re-bumps its gitlink to the
post-squash merged SHA (a squash mints a new commit) and lands next.
