# 2026.193.1015 — P14a status + OCI wire types (sdk producer half)

The sdk producer half of P14a (status subsystem + OCI registry/merge → plugins). A COMPILED-IN
`command:status` plugin (fleet fan-out + render) and a `verb:oci` plugin (the go-containerregistry
merge engine + image-user probe) cannot reach core Mechanisms — a plugin imports only the sdk — so
P14a adds the GENERIC host↔plugin / render wire types they exchange. All types are **CUE-sourced**
in `schema/status.cue` + `schema/oci.cue` and generated by `task cue:gen` (SDD — WIRE TYPES ARE
CUE-SOURCED WITHOUT EXCEPTION); there is no hand-written `*_wire.go`.

## Status wire types (`schema/status.cue` → `spec`)

- **`#DeploymentStatus`** — the rendered shape for the `charly status` table + `--json` outputs
  across every substrate (pod / vm / k8s / local / android). The `command:status` plugin fans out to
  each substrate's status-collect Op, merges the returned rows, applies the nested overlay, and
  renders. `kind` discriminates the substrate; `nested` is the RECURSIVE multi-hop child tree;
  `source` records provenance (libvirt|ledger|adb|tree|podman).
- **`#PortMapping` / `#ToolStatus`** — the structured sub-rows (`ports[]` / `tools[]`).
- **`SubstrateKind`** (hand-written `spec/status_types.go` — a distinct named string type + the 5
  substrate consts) with a CUE `#SubstrateKind` enum for wire-boundary validation (`@go(-)`, the
  `charly_names.go` scalar-enum precedent: gengotypes degrades a string enum to a plain string and
  loses the typed consts the collectors switch on).

**SDD spike findings (RDD schema spike, the evidence class the wire mandate wants):**

1. A RECURSIVE struct generates faithfully — `#DeploymentStatus.nested: [...#DeploymentStatus]` →
   `Nested []*DeploymentStatus` (gengotypes breaks the self-cycle with a POINTER slice). The JSON
   marshal is byte-identical to a value slice, so `charly status --json` parity holds.
2. JSON-tag fidelity is EXACT — required CUE fields (`kind`/`image`/`status`/`container`/`run_mode`)
   → `json:"kind"` (no omitempty); optional fields → `,omitempty`. Byte-identical to the current
   hand-written core struct → the spaced `"kind": "pod"` form the `command` check probes assert on.
   Locked by `spec/status_wire_test.go` (round-trip against the documented golden, the zero-value
   required-field-presence contract).

## OCI wire types (`schema/oci.cue` → `spec`)

- **`#MergeRequest{ImageRef, MaxMB, MaxTotalMB, Engine, DryRun}` / `#MergeReply{LayersBefore,
  LayersAfter, Skipped, Error, Notes}`** — the host↔`verb:oci` merge envelope. The plugin is
  PURE-ENGINE (go-containerregistry only; it cannot resolve box config → image ref), so the request
  carries the already-RESOLVED `ImageRef` + `Engine`; each caller resolves before invoking. P8b's
  transitional `HostBuild("merge", MergeRequest)` swaps to `InvokeProvider("verb","oci",OpMerge)`
  when the `verb:oci` plugin lands — same wire, zero reshape (R3, one shape — consumed by P8b).
  `Error` is the per-merge reply-error convention (e.g. the known podman-load EEXIST case →
  functional-but-unmerged); infra failures ride the Go error on Invoke.
- **`#ImageUserInput{Ref, UID}` / `#UserInfo{Found, Name, UID, GID, Home}`** — the base-image
  adopt-user probe (`/etc/passwd` for a uid). `InspectImageUser`'s consumer stays host-side (the
  build resolve/render half); it reaches `verb:oci` through the registry when the plugin lands.

## Scope note + R1 documentation-divergence RCA

This ships the WIRE-TYPE producer half ONLY. The two hub-dependent read helpers `ExtractMetadata` +
`ResolveNewestLocalCalVer` are DEFERRED (task P2B-completion), NOT in this PR, because they return
`BoxMetadata` — and:

**R1 RCA (documentation divergence — a doc-stale claim is an incident):** the original program plan
asserts P2 part (B) moved `BoxMetadata` + its sub-types + label consts + `CapabilityLabelMap` to
`sdk/spec`. **It never landed** — `type BoxMetadata struct` exists ONLY at `charly/labels.go:166`;
`sdk/spec` has no `BoxMetadata`. `BoxMetadata` is a ~40-field type-graph HUB referencing ~18
sub-types, of which several are owned by other in-flight cutovers (`VolumeMount`/`LabelSecretEntry`
→ P11, `CollectedAlias` → P14b). CUE-sourcing it under the R8 baked-label backward-compat constraint
(the CUE-generated struct must round-trip EXISTING baked OCI labels byte-for-byte) requires its
sub-types' json tags to be FROZEN first, so the hub migration is sequenced AFTER P11 + P14b settle
their sub-types (P2B-completion). `ParsePortMapping` (the third named helper) is ALREADY in `sdk/kit`
(core aliases it) — nothing to move. `charly box merge` CLI surface → P15's command:box (flagged,
not moved). `writeLabels` (build-time emission) → coordinated to P8b's build engine.

The consumers of these wire types — the `command:status` + `verb:oci` plugin candies and the core
rewiring/deletions — are the P14a SUPERPROJECT half, which merges after P11 + P12 (C7).
