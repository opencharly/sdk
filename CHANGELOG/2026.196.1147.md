# 2026.196.1147 — K5-Unit-1 Option A: clean the kit of the R-item, shrink the seam to 1-op

## K5-Unit-1 Option A — fix the incomplete-seam violation + shrink the DeployStateHost seam to 1-op

The K5-Unit-1 cutover (2026.196.0928) moved the deploy state-model body into
`sdk/deploykit` but introduced an **incomplete-seam violation**: the kind-blind kit
gained a deploy-kind-specific R-item (`MarshalBundleNodeLegacy` — the struct-body →
legacy-shape marshal) and the `DeployStateHost` seam threaded THREE ops that are
either kind-blind kit primitives or deploy-kind-specific transforms that never
belonged in a kind-blind kit. This cutover fixes the violation (Option A — clean the
kit now; the consumer moves to their plugin homes are the immediately-following K4
wave, tracked, not deferred).

### The seam shrinks 4-op → 1-op

`StateHostMechanisms` loses three fields; the ONE remaining field is `LoadUnifiedBundleConfig`
(the kind-blind unified-loader M, K1-gated — it dies when K1 relocates `LoadUnified` to
`sdk/loaderkit`, task #31):

- **`LatestSchemaVersion`** → collapsed to a direct `kit.LatestSchemaVersion()` call
  (a kind-blind kit primitive; the seam op was a redundant hop).
- **`AcquireDeployConfigLock`** → collapsed to a direct `kit.AcquireFileLock(path+".lock", true)`
  call (a kind-blind kit primitive; `acquireDeployConfigLock` was a thin charly alias of it).
- **`MigrateDeployEntity`** → removed (the deploy-kind-specific legacy→node-form transform
  is an R-item that contaminated the kind-blind kit; it was never the kit's to own).

### MarshalBundleNodeLegacy leaves the kit (the R-item fix)

`MarshalBundleNodeLegacy` (the deploy-kind-specific struct-body marshal with re-injected
`target:`/`nested:`/`peer:`) is REMOVED from `sdk/deploykit/deploy_file.go`. It was the
incomplete-seam violation: a kind-blind kit holding a deploy-kind-specific transform. It
returns to `charly/deploy_nodeform.go` (beside `migrateDeployEntity`, where it lived
pre-K5) as the REDESIGNED `marshalBundleNode` — the "Legacy" name AND the legacy
struct-body shape dropped, fused with `migrateDeployEntity` into ONE clean canonical
BundleNode → node-form marshal (the super leg of this cutover).

### SaveBundleConfig redesigned kind-blind

`SaveBundleConfig` is now a kind-blind file shell: it takes a `marshalNode` callback
(the deploy-kind-specific node-form serializer the caller supplies) and does ONLY the
version stamp (`kit.LatestSchemaVersion`), the `provides:` directive, the per-entry
callback invocation, and the atomic tempfile+rename write with the fail-safe re-check.
No per-kind map, no target-vocabulary switch, no deploy-kind-specific struct knowledge.
`SaveDeployState` + `CleanDeployEntry` thread the callback through to `SaveBundleConfig`
(the deploy-kind-specific marshal is the caller's responsibility, not the kit's).

### E/M justification (the 1-op seam)

`LoadUnifiedBundleConfig` is a kind-blind MECHANISM (M) — it loads ANY per-host
`charly.yml` through the unified loader and returns its `ProjectBundleConfig`, never
branching on a deploy kind. It lives in the seam (not in deploykit directly) ONLY
because the `LoadUnified` orchestration has not yet relocated to `sdk/loaderkit` (K1,
task #31): once K1 lands, plugin-bundle calls `loaderkit.LoadUnified` directly and this
seam dies entirely. Tracked K1-exit task: this seam field is deleted when K1 lands.

### K4-exit (the consumer moves)

The deploy-kind-specific marshal + the ~12 charly/ deploy-lifecycle consumers of
`SaveBundleConfig` / `SaveDeployState` / `CleanDeployEntry` remain in `charly/` core as
TRACKED K4/K5 inventory with a named K4-exit: they move to their plugin homes
(plugin-bundle / plugin-deploy-pod / plugin-check) in the immediately-following K4 wave
of this same operator window (not deferred — the K4 units begin the moment this cutover
lands). The R-item's FINAL plugin home is the K4 wave.
