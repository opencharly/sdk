# 2026.191.1530 — the check result model lands in sdk/kit (P5-unit-2)

Second unit of P5 (check-engine SDK-ization). The check engine's **result model** — the
pass/fail/skip verdict, the per-step result record, and the output formatters — now lives
ONCE in `sdk/kit`, so a plugin candy that runs a plan represents and formats results with
the same types the engine uses.

## What moved

- **`kit.Status` gains a `String()`** → `"pass"/"fail"/"skip"/"unknown"`. It was already the
  three-valued verdict enum (`StatusPass/StatusFail/StatusSkip`) a verb returns in a
  `kit.Result`; it is now also the ONE enum the engine records.
- **`kit.CheckResult`** (`checkresult.go`) — the engine's per-step record: a verb's verdict
  (Status/Message/CapturedValue) plus the bookkeeping a `Result` does not carry (`Op`, `Verb`,
  `Elapsed`, retry `Attempts`/`TotalElapsed`, `DeadlineExceeded`). `Op` is `*spec.Op`.
- **`kit.StepResult`** — one plan step's outcome (keyword/text/origin/id + its `CheckResult`).
- **`kit.FormatStepResults{Text,JSON,TAP,JUnit}`** + `kit.Plural` (`report.go`) — the check
  output reporters, moved verbatim.

## The duplication this dissolves

`kit.Status` and charly's `CheckStatus` were the **same three-valued enum**, bridged by a
`kitStatusToCheck` converter in core — the exact "mirror + convert" pattern P5 exists to
remove (kit.go's own comment read *"Status ... mirrors charly's CheckStatus ordering"* and
*"charly converts it to its internal CheckResult"*). With the result model in kit, charly's
`CheckStatus` becomes a type alias of `kit.Status` and the converter is deleted — a verb's
`kit.Result` verdict flows into a `CheckResult` with no conversion.

`kit.Result` (the verb-author verdict: what to *return*) and `kit.CheckResult` (the engine's
*record*) coexist as distinct, documented concepts — a Request/Record split, not duplication.

## Check-coverage — the formatters had NONE

`report_test.go` adds the tests the reporters never had: the exact text output (the format
the `check-live` golden oracle compares — `PASS`/`FAIL`/`SKIP` column, the `N steps:`
summary, singular/plural), the retry-info suffix (present at `Attempts>1`, absent at 1), TAP
v13 (failing point emits exactly one diagnostic block; a skip is still `ok`), a JSON
round-trip, and JUnit XML (origin grouping into suites, per-suite counts). Plus
`kit.Status.String()` (all four cases) and `kit.Plural`.
