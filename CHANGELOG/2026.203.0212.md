# 2026.203.0212 — Unit-6 sdk-contract leg (venue-descriptor seam, mechanical trio, S3b deploy-dispatch wire, cloud-init fix)

The sdk half of the Unit-6 core-minimization wave (four ordered sub-units,
one branch: `feat/unit6-seam` → `feat/unit6-mech` → merge → `feat/unit6-s3`
→ `feat/unit6-s3b`): the wire contract + shared mechanism kits that let the
superproject leg move the deploy-dispatch cluster, the step-dispatch path,
and `host_exec` out of `charly/` core into plugin candies.

## New / changed

- **S1 — the venue-descriptor `InvokeProvider` wire extension.**
  `#Executor.InvokeProviderRequest` (`schema/seam.cue`) gains a CUE-sourced
  `venue_descriptor_json` field, and `Executor.InvokeProviderOpts`
  (`executor.go`) carries the OPTIONAL extras (a `spec.VenueDescriptor`) to
  a peer-dispatch call — the zero value reproduces the pre-existing
  no-extras call shape exactly, so every existing call site is unaffected.
  Spike-proven first on a live disposable bed (`fs-spike-venue`): +48/−1
  host diff, zero deadlock across the only real broker-nesting point
  (host→peer via `inprocExecutorClient`; compiled-in venue-less callers
  reach `InvokeProvider` with zero wire hops), backward-compatible,
  descriptor round-trips losslessly.
- **`kit.VenueFromDescriptor(spec.VenueDescriptor) (spec.DeployExecutor,
  error)`** (`kit/venue_descriptor.go`, new) — re-materializes a
  `spec.VenueDescriptor` into a real `DeployExecutor` (`ShellExecutor` for
  `shell`, `SSHExecutor` for `ssh`), the plugin-side counterpart the S3b
  deploy-dispatch cluster uses to rebuild the live venue after a
  teardown-executor round trip.
- **`kit.DescriptorFromExecutor`** (`kit/venue_descriptor.go`) — the
  reverse direction, promoted during the wave's fix round to fix bug 7 (the
  nested-child venue swap dropped by the S3b port; see the superproject
  CHANGELOG's full 8-bug dossier): lets `applyParentExecOverride` capture a
  live executor back into wire form so a nested non-lifecycle child (a
  `local:` member under a `vm:`/`pod:`) can be threaded through the venue
  its parent already established, instead of falling back to the caller's
  own host executor.
- **S6 — `kit/reexec_ssh.go`** (new, +267 lines incl. test) —
  `ReexecOverSSH` relocated from `charly/host_exec.go` (which shrinks from
  273 to 45 lines). `x/term` is deliberately NOT imported here: the
  TTY-detection concern is contained in the `charly` binary's own
  `main.go` via a `wantTTY bool` parameter, so this kit stays
  dependency-clean for the ~38 candy modules that import `kit`.
- **`spec.LifecycleOptsFromEmit(EmitOpts) LifecycleOpts`**
  (`spec/deploy_wire.go`, new) — a projection function used by the ported
  `pluginDeployTarget.Add`/`.Update` to marshal deploy-apply options to the
  wire WITHOUT the live `DeployExecutor` interface field (`ParentExec`) that
  a raw `EmitOpts` marshal carries — the fix for bug 1 (a marshal crash on
  any nested-child deploy). `spec.DeployTargetUpdateOpts` (a since-unused
  parallel shape inconsistent with the pre-move design) removed.
- **`#DeployTargetDispatchRequest.host_env_json`** (`schema/seam.cue`, new
  field, `@go(HostEnvJSON, type=RawBody)`) — the marshalled `spec.HostEnv`
  (`CharlyBin`/`Home`/`Version`), computed HOST-side (the only place that
  reliably knows its own binary regardless of a substrate's out-of-process
  placement) and threaded through every lifecycle op via the dispatch
  request — the fix for bug 5 (a plugin-side `hostEnvJSON()` that was never
  actually populated in the S3b port).
- **`#DeployTargetDispatchRequest.venue_json`** (`schema/seam.cue`) — the
  wire carrier for the nested-child venue-swap fix (bug 7): threads the
  parent's `spec.VenueDescriptor` through the dispatch request so
  `applyParentExecOverride` can reconstruct it plugin-side.
- **`#VmSource.distro`** (`schema/vm.cue`) — added to the `kind:
  "cloud_image"` disjunction arm (previously only `kind: "bootstrap"`
  declared it); a closed CUE disjunction rejects unrecognized fields, so
  authoring `distro:` on a cloud_image VM source failed the WHOLE document
  load with a cascading, unrelated-looking "empty disjunction" error until
  this fix (bug 4).
- **`vmshared.effectiveDistro` + the pacman-family cloud-init path**
  (`vmshared/cloud_init_render.go`) — `RenderCloudInit` now infers `"arch"`
  when `Kind=="cloud_image" && BaseUser=="arch" && Distro==""` (explicit
  `Distro` always wins), and for any pacman-family distro (reusing the
  existing `formatForDistroID` table — arch/cachyos/manjaro/endeavouros)
  omits the `packages:` cloud-config key and instead prepends `pacman -S
  --needed --noconfirm <union>` to `runcmd`. Root cause (bug 3, "the
  wedge"): cloud-init's `packages:` directive reinstalls already-present
  Arch packages (no `--needed`), re-triggering openssh's post-install SSH
  host-key regen hook while the base image's own socket-activated sshd is
  already listening — a live key-file rewrite race that reset every SSH
  connection attempt during `kex_exchange_identification`. Isolated via an
  N=6 serial-console sampling series against a real `eval-vm` (two prior
  hypotheses ruled out first: executor threading, and `RebootStep`);
  post-fix N=6: 6/6 PASS, ~52–53s each.
- Minor `proto/plugin.proto` + `protocol/schema/plugin.cue` /
  `deploykit/steps.go` / `kit/reverse_ops.go` / `ops.go` adjustments
  supporting the above (regenerated `spec/cue_types_gen.go`,
  `proto/plugin.pb.go`).
