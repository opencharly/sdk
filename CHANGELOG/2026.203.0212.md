# 2026.203.0212 — Unit-6 S3b sdk-contract leg

The sdk half of the Unit-6 S3b implementation unit (core-minimization
program): the wire contract + shared shapes the deploy-dispatch cluster's
move to `candy/plugin-bundle` (superproject leg) needs.

## New / changed

- **`kit.VenueFromDescriptor(spec.VenueDescriptor) (spec.DeployExecutor,
  error)`** (`kit/venue_descriptor.go`, new) — re-materializes a
  `spec.VenueDescriptor` into a real `DeployExecutor` (`ShellExecutor` for
  `shell`, `SSHExecutor` for `ssh`), the plugin-side counterpart the S3b
  deploy-dispatch cluster uses to rebuild the live venue after a teardown-
  executor round trip.
- **`Executor.InvokeProviderOpts`** (`executor.go`) — carries the OPTIONAL
  extras (`VenueDescriptor`) to an `InvokeProvider` peer-dispatch call; the
  zero value reproduces the pre-existing no-extras call shape.
- **`spec.LifecycleOptsFromEmit(EmitOpts) LifecycleOpts`**
  (`spec/deploy_wire.go`, new) — a projection function used by the ported
  `pluginDeployTarget.Add`/`.Update` to marshal deploy-apply options to the
  wire WITHOUT the live `DeployExecutor` interface field (`ParentExec`) that
  a raw `EmitOpts` marshal carries — the fix for a marshal crash on any
  nested-child deploy (found live on the `check-sidecar-pod` bed; see the
  superproject CHANGELOG for the full 6-bug dossier). `spec.
  DeployTargetUpdateOpts` (a since-unused parallel shape) removed.
- **`#DeployTargetDispatchRequest.host_env_json`** (`schema/seam.cue`, new
  field, `@go(HostEnvJSON, type=RawBody)`) — the marshalled `spec.HostEnv`
  (`CharlyBin`/`Home`/`Version`), computed HOST-side (the only place that
  reliably knows its own binary regardless of a substrate's out-of-process
  placement) and threaded through every lifecycle op via the dispatch
  request, replacing a plugin-side `hostEnvJSON()` that was never actually
  populated in the S3b port.
- **`#VmSource.distro`** (`schema/vm.cue`) — added to the `kind:
  "cloud_image"` disjunction arm (previously only `kind: "bootstrap"`
  declared it); a closed CUE disjunction rejects unrecognized fields, so
  authoring `distro:` on a cloud_image VM source failed the WHOLE document
  load with a cascading, unrelated-looking "empty disjunction" error until
  this fix.
- **`vmshared.effectiveDistro` + the pacman-family cloud-init path**
  (`vmshared/cloud_init_render.go`) — `RenderCloudInit` now infers `"arch"`
  when `Kind=="cloud_image" && BaseUser=="arch" && Distro==""` (explicit
  `Distro` always wins), and for any pacman-family distro (reusing the
  existing `formatForDistroID` table — arch/cachyos/manjaro/endeavouros)
  omits the `packages:` cloud-config key and instead prepends `pacman -S
  --needed --noconfirm <union>` to `runcmd`. Root cause: cloud-init's
  `packages:` directive reinstalls already-present Arch packages (no
  `--needed`), re-triggering openssh's post-install SSH host-key regen hook
  while the base image's own socket-activated sshd is already listening — a
  live key-file rewrite race that reset every SSH connection attempt during
  `kex_exchange_identification`. Isolated via an N=6 serial-console sampling
  series against a real `eval-vm`; post-fix N=6: 6/6 PASS, ~52–53s each.
- Minor `proto/plugin.proto` + `protocol/schema/plugin.cue` /
  `deploykit/steps.go` / `kit/reverse_ops.go` / `ops.go` adjustments
  supporting the above (regenerated `spec/cue_types_gen.go`,
  `proto/plugin.pb.go`).
