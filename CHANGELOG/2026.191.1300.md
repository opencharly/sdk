## Two pre-existing bugs, found by writing the tests it never had

The 298-line lexer + recursive-descent parser had **zero test coverage** in core. Moving
untested code into the contract module would have been worse than leaving it, so the move
ships tests â€” and they immediately caught two real defects:

**1. The lexer decoded bytes as runes.** It did `r := rune(src[i])`, converting a single
BYTE to a rune, and then called the Unicode-aware `unicode.IsLetter` / `unicode.IsDigit`
on it. For any multi-byte codepoint that yields the lead byte, so a non-ASCII tag was
unusable. Against the original lexer:

```
ParseTagExpr("cafĂ©"): tag expression: unexpected character 'Â©' at position 4
```

The identifier scan advanced byte-wise too, so even where the lead byte happened to pass
`IsLetter` (Latin-1 supplement), the resulting token was a truncated byte slice that could
never match its own tag.

**2. Invalid UTF-8 matched silently.** No decode error was possible, so malformed input
lexed into garbage tokens instead of failing.

Both fixed by decoding with `utf8.DecodeRuneInString` and rejecting `RuneError` on a
1-byte advance.

## The identifier predicates stay DISTINCT â€” a correction

A first version of this change collapsed the leading-character and continuation identifier
checks into one `isTagIdentRune`, on the theory that they had "drifted". That was **wrong**,
and a `pr-validator` round rejected it. The two predicates are deliberately different:

| | accepts |
|---|---|
| leading char | `@` (sigil), letter, digit, `_` |
| continuation | the above, plus `-` `.` `:` |

`-`, `.` and `:` may appear *inside* a tag (`ci-smoke`, `v1.2`, `fedora:43`) but must not
*start* one. Collapsing them widened the grammar so `-b`, `.x`, `:y` and `a and -b` were
**silently accepted** as valid expressions instead of erroring â€” contradicting
`ParseTagExpr`'s own retained contract ("a syntax error is returned rather than silently
matching"). The stated rationale was false too: `ci-smoke` lexes because it starts with `c`,
not because the leading check was buggy.

The predicates are now `isTagIdentStart` (narrow) and `isTagIdentRune` (= start + `-` `.`
`:`), matching the original exactly, and `TestLexTagExpr_IdentifierMayNotStartWithPunctuation`
locks the contract â€” it fails against the widening.

**Behaviour parity, proven by differential.** The verbatim original lexer and the kit
implementation were run over a 36-input corpus (valid expressions, all seven syntax-error
shapes, the disputed leading-punctuation inputs, precedence cases), comparing accept/reject
*and* `Match()` results against three tag sets: **identical on every ASCII input**. The only
behaviour changes are the two UTF-8 bug fixes above.
