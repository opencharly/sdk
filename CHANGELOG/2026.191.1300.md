# 2026.191.1300 — the tag-expression filter surface lands in sdk/kit (P5-unit-1)

First unit of P5 (check-engine SDK-ization). The `--tag` / `--tag-exclude` grammar the
check engine uses to filter a plan now lives ONCE in `sdk/kit`, so a plugin candy can
filter a plan by tag with the SAME grammar rather than re-implementing it.

Moved from `charly/description_tagexpr.go` + the tag-set helpers in
`charly/description_spec.go`:

- `TagExpr`, `ParseTagExpr`, `CombineTagFilters` — the Cucumber-compatible subset
  (`and` / `or` / `not` / parens, optional `@` sigil, empty expression matches everything).
- `NormalizeTag`, `EffectiveTags` — the tag-set normalizer + de-dup.

`kit` imports only the stdlib here, so the move adds no dependency to the contract module.

## Two pre-existing bugs, found by writing the tests it never had

The 298-line lexer + recursive-descent parser had **zero test coverage** in core. Moving
untested code into the contract module would have been worse than leaving it, so the move
ships tests — and they immediately caught two real defects:

**1. The lexer decoded bytes as runes.** It did `r := rune(src[i])`, converting a single
BYTE to a rune, and then called the Unicode-aware `unicode.IsLetter` / `unicode.IsDigit`
on it. For any multi-byte codepoint that yields the lead byte, so a non-ASCII tag was
unusable. Against the original lexer:

```
ParseTagExpr("café"): tag expression: unexpected character '©' at position 4
```

The identifier scan advanced byte-wise too, so even where the lead byte happened to pass
`IsLetter` (Latin-1 supplement), the resulting token was a truncated byte slice that could
never match its own tag.

**2. Invalid UTF-8 matched silently.** No decode error was possible, so malformed input
lexed into garbage tokens instead of failing.

Both fixed by decoding with `utf8.DecodeRuneInString` and rejecting `RuneError` on a
1-byte advance. The identifier-rune predicate is factored into `isTagIdentRune` (one
definition, used by both the leading-char and continuation checks — the two had drifted:
the leading check omitted `-`, `.`, `:`).

## Check-coverage

`kit/tagexpr_test.go` — 8 tests covering normalization, de-dup + order preservation,
the empty/nil match-everything contract, operator precedence (`and` binds tighter than
`or`; `not` binds to the atom), case-insensitive keywords, identifier punctuation actually
used by tags (`fedora:43`, `ci-smoke`, `v1.2`), seven syntax-error shapes, `String()`
round-trip, and `CombineTagFilters`' four include/exclude quadrants.

Against the original byte-wise lexer, `TestLexTagExpr_UnicodeIdentifier` and
`TestLexTagExpr_InvalidUTF8` both FAIL with the verbatim errors above; they pass with the
fix. (The first attempt at that differential reverted only the outer decode and the test
still passed — a false negative. The lexer body was restored verbatim before the claim was
trusted.)
