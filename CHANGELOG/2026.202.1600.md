# 2026.202.1600 — clean --deep wire types (sdk leg)

Finalized by the fresh `pr-validator` at merge.

## What this leg is

The sdk half of the `charly clean --deep` cutover (the paired superproject leg
adds the plugin flag + the shared core purge engine): a SOLO-URGENT gap-closing
cutover adding a store-wide untagged/dangling-image purge to `charly clean`,
motivated by a real host holding 3,552 images / 132.6GB / ~97% reclaimable,
confounding build-corruption diagnosis (issue #173) and wasting the host.

### `sdk/spec/clean_wire.go`

`RetentionRequest` gains `Deep bool` (the `--deep` request flag);
`RetentionReply` gains `DeepIDs []string` + `DeepBytes int64` (the removed/
would-remove image IDs and their summed reported storage size). **R1
correction (same cutover):** the orchestrator's destructive R10 run showed
`DeepBytes` heavily overstates actual freed disk (a live purge removing 68
untagged images reported ~92.6 GiB via this sum but freed only ~4.6 GiB real
disk) — most of the summed bytes are layers SHARED with images that remain.
`DeepBytes`'s doc comment now states this plainly: it is an UPPER BOUND, not
a prediction, and pairing `--deep` with `--invalidate` gets closer to it.

### `sdk/kit/local_image.go`

`LocalImageInfo` gains a `Size int64` field, populated from the engine's
reported `"Size"` JSON field (bytes) in `ParseLocalImagesJSON` — needed to
report `--deep --dry-run`'s "reclaimable bytes" figure.

### `sdk/kit/http_fetch.go`

`humanBytes` exported to `HumanBytes` (its two existing download-progress call
sites updated in place) so `candy/plugin-clean` can format the reclaimable-
bytes report without duplicating the formatter (R3).
