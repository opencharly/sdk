# 2026.193.1222 — the command:bundle host seams + the pure deploy-state pieces (P13)

The SDK half of P13 (the `charly bundle` CLI → compiled-in `command:bundle` plugin): the
CUE-sourced wire types for the four new host-build seams, plus the genuinely SDK-portable
pure pieces of `charly/deploy.go`'s deploy-state model.

## The four seam wire types (`schema/seam.cue`)

`#DeployAddRequest` / `#DeployDelRequest` / `#DeployFromBoxRequest` / `#DeployConfigRequest`
(each with an empty `…Reply`), CUE-sourced per the SDD wire mandate, joining the P10
`#ConfigResolve*` / `#VmBuild*` family in the same file. The compiled-in `command:bundle`
plugin forwards its parsed CLI flags through these to the host, which runs the existing
deploy orchestration VERBATIM (the deploy kernel — loader, InstallPlan compiler,
`ResolveTarget`, live-executor composition — stays core, exactly as the box-build engine
stayed core behind `HostBuild("image")`). The Replies are empty: the compiled-in plugin's
`HostBuild` runs in charly's own process, so handlers print host-side and error via the
return.

`#DeployConfigRequest` carries an `Op` discriminator (show/export/import/reset/status) +
the config-mgmt subcommands' authored inputs. It is a DISTINCT seam from the entity-scoped
`#ConfigResolveRequest`/`#ConfigPersistRequest` — whole-file operator management vs
single-entity runtime-state — sharing zero handler code (R3).

## `sdk/deploykit/deploy_file.go` — the two pure state functions

`LoadDeployFile` (file → `BundleConfig`) + `RemoveBoxDeploy` (map delete) fold out of
`charly/deploy.go`. These are the ONLY zero-dependency pure pieces of the deploy-state
model; the rest (`LoadBundleConfig`/`SaveBundleConfig`/`saveDeployState`/`cleanDeployEntry`/
`MergeDeployOntoMetadata`/`ExportAllBox`) stays in charly core — it is coupled to the
unified LOADER, the process-shared deploy-config FLOCK, the legacy→node-form migrate
transform, and the runtime `Config` graph, none of which the SDK can import. Plugins reach
the ledger via the config-resolve/config-persist host SEAMS, not these functions.

## `sdk/deploykit/deploy_volume.go` — the VolumeMount resolved-state struct + pure naming

`VolumeMount` (the deploy-resolved `{VolumeName, ContainerPath}` struct) + `DeployVolumePrefix`
+ `DeployStorageDir` (the deterministic per-deploy volume-namespace naming,
`kit.ContainerNameInstance`-based) fold out of `charly/volumes.go` + `charly/deploy.go` so
`candy/plugin-deploy-pod`'s `generateQuadlet` references them (plain Go, imported).

**`VolumeMount` is plain-Go, NOT CUE-sourced — the wire mandate does not apply (R1-verified
at the marshal path).** The `ai.opencharly.volume` OCI label bytes are `[]LabelVolumeEntry
{name, path}` (encoded at `charly/generate.go`, decoded at `charly/labels.go`); `VolumeMount`
is the DEPLOY-RESOLVED form CONSTRUCTED from a `LabelVolumeEntry` + the box name at decode
(`VolumeName = "charly-<box>-<name>"`), and is NEVER marshaled anywhere. It is resolved
runtime state — the same category as `ResolvedBindMount` — so `deploykit` plain-Go is its
correct permanent home. (An earlier spec-CUE detour was reverted: `BoxMetadata` is the
resolved aggregate, never whole-marshaled, so the `BoxMetadata`→spec fold (#23) CUE-sources
the `Label*` WIRE types, not the resolved `VolumeMount` — its home follows `BoxMetadata`'s
own runtime classification.)

`ResolvedBindMount` (the same resolved-state category) stays in `charly/enc.go` — another
cutover's single-owner file (C6) — and relocates to deploykit with that enc move.
`ResolveVolumeBacking` + `resolveVolumeHostPath` stay in charly (coupled to the enc.go
encryption-naming cluster); `scopeVolumesToDeployKey` stays (reads `*BoxMetadata`). They
fold with their owning cutovers.
