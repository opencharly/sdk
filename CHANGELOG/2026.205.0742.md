# 2026.205.0742 — sdk leg: new `#PodConfigProjectVolumeRequest`/`Reply` + `#PodConfigSetupRequest.host_env_json` wire seams

The sdk contract-repo leg of the deploy-volume-persistence fix (companion
`opencharly/charly` PR). Two additive, non-breaking wire-type additions —
internal host↔plugin seam types, never authored `charly.yml` surface, so
**no `#SchemaVersion` bump**.

## `#PodConfigProjectVolumeRequest` / `#PodConfigProjectVolumeReply`

New seam types (`schema/seam.cue`) backing a narrow `pod-config-project-volume`
host builder: given a deploy's `box`/`instance`, the host resolves the SAME
merged project+per-host-overlay tree `charly bundle add` already walks
(`resolveTreeRoot`) and returns only the resolved entry's `Volume` field.
Lets `candy/plugin-deploy-pod`'s Setup body fall back to a deploy's
PROJECT-declared `volume:` override (e.g. a disposable check bed's
`volume: [{type: encrypted}]`) when neither a CLI flag, the
`CHARLY_VOLUMES_<BOX>` env var, nor the per-host overlay already carries one
— a gap that silently dropped the declaration and left the encrypted bind
mount never established.

## `#PodConfigSetupRequest.host_env_json`

New optional field on the existing `#PodConfigSetupRequest`, mirroring
`#DeployTargetDispatchRequest.host_env_json` (`#HostEnv`: `CharlyBin`/`Home`/
`Version`). Setup's quadlet emission needs the REAL charly binary path for
the encrypted-mount `ExecStartPre=` line
(`deploykit.QuadletConfig.CharlyBin`) — `os.Executable()` called from inside
the out-of-process `candy/plugin-deploy-pod` plugin resolves to the PLUGIN's
own binary, not the charly CLI (the same bug class `unified_targets.go`'s
`hostEnvJSON()` already fixed for the lifecycle-Op family, R10 bed-found bug
#5 / S3b). The companion `charly` PR populates this field with the SAME core
`hostEnvJSON()` helper and threads the decoded value through Setup.
