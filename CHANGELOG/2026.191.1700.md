# 2026.191.1700 — the check-variable expansion grammar lands in sdk/kit (P5-unit-4)

Fourth unit of P5 (check-engine SDK-ization). The `${NAME[:arg]}` check-variable expansion
grammar moves from `charly/checkspec.go` into `sdk/kit/checkvars_expand.go`, so a plugin candy
that runs a plan expands variables with the SAME grammar the check engine uses:

- **`kit.ExpandTestVars`** / **`kit.TestVarRefs`** — substitute / enumerate `${NAME}` and
  `${NAME:arg}` references (the `HOST_PORT:6379` parameterized form).
- **`kit.IsRuntimeOnlyVar`** + **`kit.RuntimeOnlyVarPrefixes`** — classify a var key as
  runtime-only (a build-scope check may not reference `HOST_PORT`/`VOLUME_PATH`/`CONTAINER_IP`/
  `DEPLOY_NAME`/`HOST`/…).
- **`kit.ExpandOpVars`** — expand every `${...}` on a `spec.Op` in place (its `StringFields()`
  plus the opaque `PluginInput` map).
- **`kit.ExpandAnyVars`** / **`kit.CollectAnyStrings`** — the depth-first expand / read-only
  collect over a plugin-input value (scalar / nested map / list).
- **`kit.TestVarRefPattern`** — the shared `${…}` regex.

Only the vocabulary-INDEPENDENT expansion moves. The check SEMANTICS that consult the core
`VerbCatalog` — `opEffectiveDo`, `opEffectiveContexts`, `opActsInBuildDeploy` (the last still
type-asserts `*grpcProvider`) — stay in `charly/checkspec.go`.

## Check-coverage

The pure tests move with the code (`checkvars_expand_test.go`): the `${NAME[:arg]}` substitution
+ unresolved-ref reporting (order-preserving, deduplicated), the parameterized-vs-plain key form,
the runtime-only classification (incl. `DEPLOY_NAME`/`HOST`), and — new coverage the originals
lacked — the nested-map/list `ExpandAnyVars` walk (non-string scalars pass through; a nested
`${MISSING}` is reported) and `CollectAnyStrings`. The core `Op`-integration tests
(`TestCheck_ExpandVars`) stay in core and exercise the `ExpandOpVars` alias.
