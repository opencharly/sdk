# 2026.204.1329 — K1-unblock wave 1: `#ResolvedProject.resources` + the arbiter's HostArbiter retirement

The first wave of the K1-UNBLOCK program: the resource arbiter's last 2
genuinely LoadUnified-coupled host dependencies (gather/resources) move off
the bespoke `ExecutorService.HostArbiter` reverse RPC onto the SAME generic
`HostBuild("resolved-project")` envelope every other resolved-project
consumer already reads — closing the arbiter's K1 residue entirely and
retiring a whole RPC method + its request/reply wire types.

## What changed

- **`#ResolvedProject.resources`** (`schema/resolvedproject.cue`) — a new
  `map[string]#ResolvedResource` field, filled host-side exactly like the
  existing `deploy` field (a straight assignment from the project's resolved
  `resource:` kind entities). One-line, SDD-clean addition; `ResolvedResource`
  was already CUE-generated.
- **`sdk/deploykit/preempt_resolve.go`** (new) — the portable, LoadUnified-free
  resource-arbiter deploy-tree helpers extracted from charly core's former
  `preempt.go`: `MergedDeployTree` (project ⊕ per-host overlay, via the
  already-portable `LoadDeployConfigForRead`/`MergeBundleNode`),
  `HolderAddrFor` (reads the node's stamped `Descent.Venue` instead of a
  fresh registry lookup — a pure function of already-materialized data),
  `FilterPreemptibleHolders`, `FindVMClaimant`, `GpuVendorTokens`. Full test
  coverage in `preempt_resolve_test.go`.
- **`ExecutorService.HostArbiter` reverse RPC — RETIRED entirely**
  (`protocol/schema/plugin.cue`, `task wire:gen` regenerated): the `gather`/
  `resources` actions it carried are the arbiter's last 2 K1-blocked reads;
  now that they're servable off `HostBuild("resolved-project")`, the whole
  bespoke RPC (and its `HostArbiterRequest`/`HostArbiterReply` messages) is
  dead weight, not a kept exception (R5). `sdk.Executor.HostArbiter(...)`
  client method deleted (`executor.go`). `#ArbiterGatherReply`/
  `#ArbiterResourcesReply` CUE defs deleted (`schema/arbiter.cue`);
  `ArbiterSeamGather`/`ArbiterSeamResources` consts deleted
  (`spec/arbiter_consts.go`). `#HolderAddr`/`#HolderDescriptor`/
  `#PreemptLease`/`#PreemptLedger`/`#ArbiterInvokeInput`/`#ArbiterInvokeReply`
  are UNCHANGED (still used by the 6 InvokeProvider-dispatched seams +
  the in-core proxy).
- **`internal/wiregen/main_test.go`** — the retired `HostArbiterRequest` doc
  expectation removed from `TestProtocolDocsComplete`'s fixed table.

## Consumer-side companion (superproject, same wave)

`candy/plugin-preempt/arbiter.go`'s `hostGather`/`hostResources` now call
`HostBuild("resolved-project")` + the new portable deploykit helpers instead
of `Executor.HostArbiter(...)`; `charly/arbiter_host.go` (the host-side RPC
handler, now unreachable) is deleted; `charly/preempt.go`'s
`gatherDeployNodes`/`gatherPreemptibleHolders`/`lookupVMClaimant`/
`holderAddrFor`/`sortedHolderKeys` (now unreachable, superseded by the
portable helpers) are deleted; `host_build_config_resolve.go`'s VM-claimant
lookup repoints to the same portable `deploykit.FindVMClaimant` (R3 — one
implementation, not two). See the superproject's own CHANGELOG entry for the
full consumer-side detail.
