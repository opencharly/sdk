# 2026.201.1424 — Store-global build lock: serialize whole-image builds against the shared podman store

`kit.AcquireStoreBuildLock` / `kit.StoreBuildLockPath`: a blocking,
user-cache-scoped file lock (`~/.cache/charly/locks/store-build.lock`) that
serializes whole-image-build invocations against the shared
containers-storage. podman's rootless store (buildah, transient_store
enabled) is NOT safe for two concurrent multi-stage builds: under a two-bed
concurrent roster it produced a buildah nil-pointer panic in
`COPY --from=<stage>`, a mid-build container rootfs missing `/bin/sh`, and
`replacing mount point: file exists` on build-container teardown
(opencharly/charly#149). Serial builds pass identically every time.

The store is therefore treated as a single-writer resource: the whole
`charly box build` image set takes this lock (acquired in
candy/plugin-build's `runBoxBuild`, held across `buildImages` only —
resolve/render mutate no store state), while non-build phases across
processes (deploy/exec/check) stay parallel. Resource-token arbitration over
the store, not a hidden retry or serialization of a parallel-safe contract.

Verified: `TestStoreBuildLockPath` (path shape + ErrLockBusy contention +
reacquire-after-release); full standalone battery green; live proof: two
concurrent beds that previously corrupted each other's builds now pass the
build phase with the lock held.
