# 2026.205.1055 — add_candy-on-pod fix, companion 1/3: CheckBedReply.HasAddCandy

Companion schema change to the `charly` superproject cutover "make add_candy-on-pod
work end-to-end" (a severe, pre-existing, never-before-exercised bug: no disposable
bed in the repo ever combined a `pod:` substrate with `add_candy:`, so the overlay
build's ctx-threaded plans, the resolved-image discriminator, and the check-bed
runner's `--tag` handling had never been exercised live together — all three broke
the moment a real bed exercised them).

## What changed

`#CheckBedReply` (`sdk/schema/seam.cue`) gains `has_add_candy?: bool @go(HasAddCandy)`
— the host-side check-bed setup op now reports whether the bed's node carries a
non-empty `add_candy:` list. `candy/plugin-check`'s bed runner uses this to decide
whether its `config`/`start` steps may safely pass `--tag <base-build-tag>` (a
non-overlay bed's freshness proof) or must omit it (an overlay bed's freshness proof
is the just-built overlay tag, resolved via `resolveDeployResolvedImage` — passing
`--tag` would force `config`/`start` to redeploy the un-overlaid base image instead,
silently dropping every `add_candy:` candy from the running container).
