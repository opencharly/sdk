# 2026.191.1837 — the plan-execution helpers land in sdk/kit (P5-unit-3)

Third unit of P5 (check-engine SDK-ization). Three self-contained plan-execution helpers move
from charly core into `sdk/kit`, so a plugin candy that runs a plan shares them with the check
engine instead of re-implementing them:

- **`kit.RunWithEventually`** (`eventually.go`) — the `eventually:` retry loop + the
  killed-probe bounded re-attempt (a probe SIGKILLed by a transient host-load spike is re-run;
  a probe that ran and failed is not). Uses the already-shared `kit.SignalKillErrMarker`.
- **`kit.ScenarioContext`** (`scenario.go`) — the per-plan-run mutable state: the `capture:`
  store, the `${STEP_ID}` thread, and the background-PID reaper. Self-contained (`sync` only).
- **The acceptance-depth ladder** (`checklevel.go`) — `CheckLevel{None,Build,NoAgent,Agent}`,
  `DefaultCheckLevel`, `ResolveCheckLevel`, `IsValidCheckLevel`, `CheckLevelReaches`,
  `CheckLevels`.

`plan_validate.go` and `check_substitute.go` deliberately **stay in core**: the former is
invoked by both `charly box validate` and the `charly feature` HostBuild seam (its own header
documents this); the latter is bound to the core CUE loader (`decodeEntityViaCUE`).

## Check-coverage

The tests move with the code: `eventually_test.go` (the killed-probe retry semantics — a probe
killed on every attempt fails loudly after the cap; a ran-and-failed probe is not retried) and
`checklevel_test.go` (the ladder ordering + default + validity, which are load-bearing for how
deep every bed runs). The core `ExtractMetadata` label round-trip stayed in core (it exercises
the OCI label surface, not the ladder).
