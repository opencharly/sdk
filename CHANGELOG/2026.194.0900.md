# 2026.194.0900 — VmDomainIdentity: the per-deploy VM domain identity (sdk producer, #33)

The sdk producer half of #33 (VM naming: the libvirt domain is keyed by the DEPLOY, not the shared
`kind:vm` entity, so sibling beds referencing one entity are collision-free by construction). The
consumer (charly core + `candy/plugin-vm` + `candy/plugin-deploy-vm`) lands in the superproject; its
VM-bed R10 (`check-builder-vm` ∥ `check-substrate` → distinct domains + independent overlays) is the
gate.

- **`vmshared/vm_helpers.go` — `VmDomainIdentity(deployName)`** — normalizes a deploy/bundle name into
  the per-deploy VM DOMAIN IDENTITY that keys the libvirt domain (`charly-<identity>`), the per-domain
  state dir, the managed ssh alias, and the ssh-port ledger entry. Strips a leading `vm:` and flattens
  the instance (`/`) and nested-path (`.`) separators to `-`, so a bare bed name maps to itself
  (`check-builder-vm` → `check-builder-vm`), a bundle ref maps to its VM token (`vm:arch` → `arch`), and
  a direct `charly vm create <entity>` (whose domain identity IS the entity) is unchanged. Both the host
  preresolver and `candy/plugin-deploy-vm` call it on the SAME deploy name, so the domain they derive
  always agrees. Pinned by `vmshared/vm_domain_identity_test.go` (`TestVmDomainIdentity`), including the
  load-bearing property that two sibling beds sharing one entity map to DISTINCT identities.
- **`vmshared/vm_helpers.go` — `VmDiskDir` doc** — clarified that the per-VM disk dir
  (`output/qcow2/<entity>`) is now the shared read-only BASE every per-deploy overlay backs onto, keyed
  by the disk SOURCE (the `kind:vm` entity).
- **`spec/deploy_wire.go` — `LifecyclePrepareInput` field comments** — `SSHPort` / `Alias` / `StateDir`
  now key off the per-deploy domain identity (`resolveVmSshPort(spec, domainIdentity)`,
  `VmSshAlias(domainIdentity) = charly-<deploy>`, `~/.local/share/charly/vm/charly-<domainIdentity>`);
  `Entity` still names the disk/spec source (`node.From`-resolved). Comment-only — hand-written wire
  type, no generated-file drift.
- **`deploykit/deploy_tree.go` + `deploykit/deploy_chain.go` — the nested-child SSH alias derivation
  (the seam the concurrent VM-bed R10 caught).** `VmChildExecutor` and `AppendHopForFlatPath`'s `ssh`
  hop derived the alias from `node.From` (the shared `kind:vm` ENTITY) → `charly-<entity>`, so a nested
  `local:` child under a `vm:` bed SSHed to `charly-eval-vm` instead of the bed's live per-deploy domain
  `charly-<bed>` (both beds died at `deploy nested local child` with `Could not resolve hostname
  charly-eval-vm`). Both now derive `SSHParamsForVm(vmshared.VmDomainIdentity(<deploy path>))` — the
  SAME per-deploy domain the vm lifecycle named (`vmDomainIdentity` flattens a dotted member path
  identically), so sibling beds sharing one entity get distinct, resolvable aliases. `VmChildExecutor`
  drops its now-unused `node` param (the one caller + the charly alias updated); `SSHParamsForVm`'s
  param is renamed `domainID`. The two inverted docstrings corrected (R5). Pinned by
  `deploykit/deploy_tree_test.go` (`TestVmChildExecutor_PerDeployAlias`) — asserts the per-deploy alias
  + the sibling-distinct collision-free property; it fails against the old entity-keyed derivation. This
  seam relocated into sdk/deploykit in P13, outside the P33 charly/plugin sweep surface — R10's
  concurrent bed found what the static grep could not.
