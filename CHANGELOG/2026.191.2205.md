# 2026.191.2100 — the check-engine plan walk lands in sdk/kit (P5)

The check engine's plan walk dissolves out of charly core into `sdk/kit`
(`planrun.go` + `planspec.go`), so any plugin candy that runs a plan drives the
SAME walk the check engine uses.

- `kit.RunOne` — the per-op walk: skip:/exclude_distros:/context-vs-mode
  handling, `${VAR}` expansion, venue swap, do-mode routing, the `eventually:`
  retry, and verb dispatch through the seam.
- `kit.RunPlan` / `runUnit` — flatten a `LabelDescriptionSet` by layer, gate step
  selection by mode (verify-only skips mutating steps; feature-run skips
  deterministic run: steps), route agent steps to the grader, dispatch run:/check:
  through `RunOne`.

Three kind-blind host seams (no concrete kind / provider word / `ExecContext`
crosses them):

- `kit.PlanContext` — the driver surface (mode, distros, env, venue swap, the
  per-verb `EffectiveDo` + `ContextSkipReason` the core `VerbCatalog` grammar
  computes).
- `kit.VerbResolver` — the verb-dispatch seam (satisfied by the core provider
  registry): `RunVerb` + `RunProvisionAct`.
- `kit.StepGrader` — the agent-step grader.

Also moved (the plain data + pure helpers the walk consumes): `LabeledDescription`
/ `LabelDescriptionSet` (+ `IsEmpty`), the `DoMode` enum, `GraderRequest`, and
`KeywordOf` / `StepDoMode` / `StepID` / `EffectiveStepID` / `FilterHostVars`.

The host-serving atoms (endpoint/venue/container resolution, HTTP-do, the
provider registry, the CUE loader) stay in core behind these seams; the
`VerbCatalog` grammar + `opActsInBuildDeploy` (registry-coupled) and the
loader/harness-coupled check files stay core too — they move with the loader (P6)
/ harness (P12).
