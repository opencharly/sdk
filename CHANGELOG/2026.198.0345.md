# 2026.198.0345 — K4 shared-resolver extraction (the pod-deploy + check-harness broker set)

The K4 pod-deploy cutover (`charly` repo, moving `config_image.go` +
`pod_lifecycle_resolve.go`'s deep-body into `candy/plugin-deploy-pod`) and the
K5 check-harness cutover (moving `check_members.go`/`check_cmd.go`/
`host_build_check_run.go`/etc into `candy/plugin-check`) share a set of 9
resolver primitives that neither cone can leave core without. This is the
forced sdk producer leg both consume — landed first, standalone, so K5 isn't
blocked waiting on K4's full cutover (config_image.go's seam design + the R10
gate) before it can even start rebasing.

## Change

Split between `sdk/kit` and `sdk/deploykit` per the kit-cannot-import-deploykit
constraint (deploykit already imports kit):

- **sdk/kit**: `IsHostNetworked`/`ContainerRunning` (`container_probe.go`),
  `ResolveBoxName` + its ref-parsing helpers `StripURLScheme`/`ParsedRef`/
  `IsRemoteImageRef`/`ParseRemoteRef` (`remote_ref.go`), `ExtractMetadata`/
  `InspectLabels` (`box_metadata.go` — ~330 lines, fully portable: every
  dependency resolves to stdlib/kit/spec, no project-loader touch anywhere),
  `SidecarContainerName`/`SidecarContainerNameInstance`/`PodName`
  (`sidecar_naming.go`), and the plugin-primary registry + `ResugarPlan`
  (`plugin_primary.go` — needed by `MarshalBundleNode`'s plan-step round-trip).
- **sdk/deploykit**: `ResolveContainer`/`ResolveSidecarContainer`
  (`container_resolve.go` — need `ResolveBoxEngineForDeploy`'s
  `LoadDeployConfigForRead`), `MarshalBundleNode` + its `bundleDiscForEntity`/
  `scalarFieldValue` helpers (`deploy_nodeform.go` — needs `SortedNestedKeys`),
  and a new `ResolveDeployBoxName(key, instance, *spec.ResolvedProject)`
  (`deploy_box_name.go`) — a portable twin of charly-core's existing
  `LoadUnified`-based resolver, for a caller (a plugin) that already holds a
  `ResolvedProject` envelope instead of a project loader. charly-core's own
  resolver is unchanged — a cheap parse-only lookup every current caller,
  including the check harness, already uses; the two forms share identical
  precedence (per-host deploy-config overlay wins, then the project-level
  fallback, then the bare key) and must not diverge.

The existing charly-core function bodies (`resolveContainer`, `resolveBoxName`,
`ExtractMetadata`, `marshalBundleNode`, etc.) are intentionally UNCHANGED —
both K4's pod files and K5's check-harness files still call them unqualified
today. They become dead once BOTH cutovers land and are deleted then, as the
extraction's own R5 completion (tracked as a follow-up, not deferred
indefinitely).

Test coverage migrated alongside the code it protects: `TestBundleDiscForEntity_PodWorkloadNotGroup`
→ `sdk/deploykit/deploy_nodeform_disc_test.go`; the `ExtractMetadata` label
round-trip suite (check_level, env-object decode, singular-label contract) →
`sdk/kit/box_metadata_test.go`.

Verified: `go build`/`vet`/`test`/`golangci-lint` clean (0 issues) on the whole
sdk module.

## Follow-up: R3 duplicate dissolved during PR validation (fully tested and validated)

`kit/box_metadata.go`'s `InspectLabels` var originally defaulted to its own
`defaultInspectLabels` — a byte-identical duplicate of `kit/local_image.go`'s
`InspectImageLabels`, which the parallel K3 leg (sdk#65, merged earlier the
same day) had already promoted from the same charly-core source
(`charly/labels.go`'s `defaultInspectLabels`). Two independent extractions of
the same function, racing each other. Fixed by repointing `InspectLabels` to
the canonical `InspectImageLabels` directly and deleting the now-dead
duplicate body (`os/exec`/`strings` imports dropped with it, unused
elsewhere in the file). `kit/box_metadata_test.go`'s stubbing tests are
unaffected — they only swap `InspectLabels` for a fake, never call the
default implementation by name.

Also confirmed a `golangci-lint` run reporting 12 issues against paths in
OTHER worktrees (one already removed, one from a concurrent session) was a
stale-cache artifact, not a real finding — `golangci-lint cache clean`
resolved it to 0 issues.
