# 2026.201.0757 — Repo cache freshness for mutable refs (@main staleness root fix)

A cached remote-repo export was reused forever once downloaded: `DownloadRepo`
checked only that the cache directory existed. A mutable ref (a branch such as
`main`, or an unversioned ref resolved to the default branch) therefore froze
at its first-download content — a pre-#146 `charly@main` export kept serving
protocol-v1 builder-plugin sources indefinitely, and every box build with an
unversioned builder-plugin ref failed `Plugin version: 1, Client versions:
[2]` (opencharly/charly#158) even after the box repos re-pinned their candies
to the protocol-v2 tag.

`kit.DownloadRepo` now records the clone's resolved commit in a provenance
sidecar (`<cache>.ref`) and reuses the cache ONLY when that provenance equals
the ref's currently resolved commit — a moved branch is re-downloaded instead
of silently serving stale content. The freshness check costs the one
`ls-remote` `GitResolveRef` already performed; immutable coordinates (tags,
SHAs) always match, so their cache-hit behavior is unchanged. The
download/refresh is serialized per cache path (`<cache>.lock`) and published
by a temp-clone + rename swap, so readers see a complete old or new tree,
never a half-clone. Caches written before this contract have no sidecar and
self-heal on first access. `kit.IsMutableRef` is the shared
mutable-vs-immutable classifier the host uses to decide when to delegate.

Verified: `TestDownloadRepoFrom_RefreshesMovedBranch` (local file:// remote —
branch advance re-downloads, unmoved ref hits, no swap residue),
`TestRepoCacheFresh`, `TestIsMutableRef`; full standalone battery
(`GOWORK=off` tidy no-op, verify, build, vet, test) green. Live end-to-end
(with the matching charly core delegation): a stale `charly@main` export
self-healed and `charly -C box/arch box build arch-test` passed.
