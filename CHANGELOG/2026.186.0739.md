# kit: ssh-config + guest-readiness waits + charly-delivery helpers (vm lifecycle externalization)

The generic host-surface machinery the externalized vm deploy plugin needs moves into sdk/kit (kit stays
STDLIB + spec only ‚Äî the poll/readiness subsystem is NOT imported; see the injected-poll note):

- `kit/sshconfig.go` ‚Äî the managed ssh-config Host-stanza machinery (VmSshStanza / WriteVmSshStanza /
  EnsureSshConfigInclude / RemoveVmSshStanza / RemoveSshConfigInclude / ListVmSshAliases / VmSshAlias),
  moved from `charly/ssh_managed_config.go`. `kit/profile.go` gains `ReplaceOrPrependManagedBlock` +
  `ManagedBody` (the ssh Include line must PREPEND, outside any Host block).
- `kit/ssh_wait.go` ‚Äî `SSHArgs` + `WaitForSSH`/`WaitForCloudInit`/`WaitForPackageLock`, host-surface
  os/exec ssh with the exact load-bearing tuning preserved. To keep kit free of the heavy readiness/poll
  subsystem (vmshared ‚Üí x/crypto), the waits take an INJECTED `PollFunc`/`PollCond`: the caller (core or
  the vm plugin, which legitimately have the readiness subsystem) wraps its own pollUntil + resolved
  bounds. kit owns the ssh mechanics; the caller owns the readiness policy. One poll primitive (R3).
- `kit/charly_install.go` ‚Äî `EnsureCharlyInVenue` (generic, over kit.DeployExecutor) + `EnsureCharlyInGuest`
  (host-surface ssh/scp) + `HostCharlyIsNewer` (the single CalVer arbiter), sharing ONE probe‚Üícompare‚Üí
  deliver decision (R3), moved from `charly/charly_install.go`. `os.Executable()` ‚Üí a `charlyBin` param
  (HostEnv.CharlyBin ‚Äî the plugin's own executable is the PLUGIN binary).
- `spec/deploy_wire.go` ‚Äî `HostEnv.Version` (the host CalVer) + `LifecyclePrepareInput` (the resolved
  kind:vm entity + ssh coords + prior VmDeployState the host prepare hook ships to OpPrepareVenue).

R10: check-charly-vm PASS (steps=6) + check-k3s-vm PASS (steps=6, 112s) ‚Äî the externalized vm plugin
drives the whole venue lifecycle over these kit helpers on fresh rebuilds.
