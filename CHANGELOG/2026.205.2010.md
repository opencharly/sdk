# 2026.205.2010 — K1-alpha regression fix: --add-candy ref threaded into the resolved-project envelope

Companion to the `charly` superproject's K1-alpha regression fix (task #13): `charly
bundle add --add-candy <ref>` / a deploy's `add_candy: [<ref>]` failed with
`bundle compile: candy "<name>" not in resolved-project envelope (order=[...])`
for a REMOTE candy ref (e.g. `@github.com/opencharly/charly/candy/pod-addcandy-marker:vTAG`)
— any candy never reachable from a box's own image closure.

## Root cause

`candy/plugin-bundle`'s compile handler (`compileDeployPlans`) re-fetches the
resolved-project envelope independently via `HostBuild("resolved-project")`, using
only the project directory. The HOST's own pre-compile scan
(`scanCandiesForRef`) already widens its candy map for a remote add-candy ref via
a synthetic-augmented `ScanAllCandyWithConfig` call, but that widening was never
threaded into the plugin's separate envelope re-fetch — so a remote add-candy
resolved host-side (used to compute the compile `order`) was completely absent
from `rp.Candies`/`rp.CandyModels`, and `BuildDeployPlan`'s per-name lookup failed.

## What changed

- `#ResolvedProjectRequest` (`schema/resolvedproject.cue`) gains
  `extra_candy_refs?: [...string] @go(ExtraCandyRefs)` — mirrors
  `ResolveOpts.ExtraCandyRefs`'s existing "candy refs to collect IN ADDITION to the
  image-closure walk" contract (used elsewhere for a check bed's own `add_candy:`),
  now reachable from a `resolved-project` HostBuild caller too.
- `#DeployCompileRequest` (`schema/seam.cue`) gains the same
  `extra_candy_refs?: [...string] @go(ExtraCandyRefs)` field, so the charly-side
  compile caller can thread the add-candy ref it resolved host-side through to the
  plugin's `OpCompile` request.
- `spec/cue_types_gen.go` regenerated via `task cue:gen` (clean diff — only the
  two intended fields).
