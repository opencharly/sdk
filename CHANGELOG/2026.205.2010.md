# 2026.205.2010 — K1-alpha regression fix: --add-candy refs threaded into every independent resolved-project envelope fetch

Companion to the `charly` superproject's K1-alpha regression fix (task #13):
`charly bundle add --add-candy <ref>` / a deploy's `add_candy: [<ref>]` failed
for a REMOTE candy ref (e.g.
`@github.com/opencharly/charly/candy/pod-addcandy-marker:vTAG`) — any candy
never reachable from a box's own image closure. Two SEPARATE errors, from two
INDEPENDENT resolved-project envelope fetch sites the charly-side compile and
overlay-build paths each own:

1. `bundle compile: candy "<name>" not in resolved-project envelope
   (order=[...])` — the STANDARD `--add-candy` compile path.
2. `OCITarget.Emit(<name>): candy "<name>" not found` — the add_candy-ON-POD-
   OVERLAY deploy path (a deploy whose deploy-key differs from its image-name,
   e.g. `check-addcandy-pod`).

## Root cause

Both `candy/plugin-bundle`'s compile handler (`compileDeployPlans`) and
`candy/plugin-installstep`'s `getGenerator` (used to render HOST-COUPLED
build-emit words during an overlay build) re-fetch the resolved-project
envelope INDEPENDENTLY via `HostBuild("resolved-project")`, using only the
project directory. The HOST's own scans that computed the compile `order` /
the overlay `Generator` already widen their candy maps for a remote
add-candy ref (`scanCandiesForRef`'s synthetic-augmented scan;
`hostBuildOverlay`'s `ResolveOpts.ExtraCandyRefs`) — but neither widening was
ever threaded into these plugins' SEPARATE envelope re-fetches, so a remote
add-candy candy was completely absent from both `rp.Candies`/`rp.CandyModels`
and the overlay-side `getGenerator`'s own re-fetch.

## What changed

- `#ResolvedProjectRequest` (`schema/resolvedproject.cue`) and
  `#DeployCompileRequest` (`schema/seam.cue`) both gain
  `extra_candy_refs?: [...string] @go(ExtraCandyRefs)` — mirrors
  `ResolveOpts.ExtraCandyRefs`'s existing "candy refs to collect IN ADDITION
  to the image-closure walk" contract (already used elsewhere for a check
  bed's own `add_candy:`), now reachable from a `resolved-project` HostBuild
  caller and a compile caller.
- `#BuildEnv` (`schema/buildwire.cue`) gains the SAME
  `extra_candy_refs?: [...string] @go(ExtraCandyRefs)` field — the wire
  struct the host already puts in `op.Env` for an OpEmit Invoke — so the
  overlay-build's host-coupled step-emit dispatch (charly's
  `dispatchOCIStep`) can thread the host's own already-widened candy set
  through to `plugin-installstep`'s `getGenerator`.
- `spec/cue_types_gen.go` regenerated via `task cue:gen` (clean diff, three
  intended fields only).
