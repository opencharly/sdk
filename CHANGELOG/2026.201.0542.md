# 2026.201.0542 — F-CLI-NEST: declared subcommand catalogs for command-class plugins

`ProvidedCapability` gains an optional `subcommands` field (`repeated CLISubcommand`,
name+help pairs) a `class="command"` capability can declare over Describe. The SDK
exposes it as `sdk.ProvidedCapability.Subcommands []sdk.CLISubcommand`, threaded
through `BuildCapabilities`, plus a new `sdk.KongSubcommands(v any) []CLISubcommand`
reflection helper that derives the catalog directly from an existing Kong-tagged
grammar struct (a field's `name:`/`help:`/`hidden:` tags, falling back to Kong's own
kebab-cased field name when `name:` is absent — RDD-spiked live against `kong.New`
to confirm a value-carrying `cmd:"<value>"` tag is NEVER read as a name, only its
presence matters).

This is additive and backward-compatible: a capability declaring no subcommands
keeps today's flat pass-through Args grammar byte-for-byte.

## Motivation

Consumed by the paired charly-repo cutover (`fix(command): restore nested CLI
grammar for check/box-list — F-CLI-NEST`), which fixes two pre-existing main
regressions surfaced by the `check-charly-selftest-pod` bed: `charly check --help`
showing only a generic out-of-process placeholder instead of its real subcommand
list, and the `box.list.boxes` MCP tool having silently disappeared — both traced
to the SAME root cause (every command-class plugin's Kong grammar collapses to an
opaque pass-through node, invisible to `charly __cli-model`/MCP tool generation,
once its CLI moves off a static core Kong struct onto dynamic dispatch).
