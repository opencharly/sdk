# 2026.195.0300 — in-proc CLI dispatch primitive (#52) + the pod-lifecycle wire carriers (K4, sdk half)

The sdk contract half of the K4 deploy-lifecycle deep-body move (superproject half in that repo's
CHANGELOG). Two additions, both purely additive: the shared in-process CLI-dispatch primitive that
fixes the #52 subcommand-`--help` fall-through across every command plugin, and the two CUE-sourced
wire carriers the pod plugin's lifecycle Ops resolve into.

## What's new — `sdk/clidispatch.go` (+ `sdk/clidispatch_test.go`): the #52 fix at the shared root

Every compiled-in command plugin (`command:alias` / `command:bundle` / `command:check` / `command:box`
/ `command:migrate` / `command:status` …) dispatches its own kong grammar IN-PROCESS. kong calls
`os.Exit` on `--help` / `--version` / a parse error — which, in-process, would kill the whole `charly`
binary. Worse, a subcommand `--help` (`charly alias set --help`) fell through kong's exit path and
surfaced as **exit 80** instead of a clean 0. The fix is ONE shared primitive (R3 — a single
dispatch root, not a per-plugin patch):

- **`kongExitPanic{code}`** + a `kong.Exit(func(code int){ panic(kongExitPanic{code}) })` binding — the
  panic-sentinel that converts kong's `os.Exit` into a recoverable panic, so an in-proc CLI never
  tears down the host process.
- **`RunInProcCLI(name, cli, args, …kong.Option) error`** — parse + run a leaf command; recovers the
  sentinel, returning `nil` for a clean `--help`/`--version` (code 0, no Run) and `*ExitCodeError`
  for a non-zero exit (propagated to the process exit via `main`'s `errors.AsType` mapping).
- **`ParseInProcCLI(…) (done bool, err error)`** — parse-only variant for plugins that need the parsed
  context back; `done=true` signals a handled `--help`/`--version` (caller returns without running).

`clidispatch_test.go` ships 8 regression tests that would FAIL without the fix: leaf `--help` → 0 with
no Run, top-level `--help` → 0 with no Run, a real invocation runs, a leaf `*ExitCodeError{Code:7}`
propagates, a parse error surfaces, and the `ParseInProcCLI` done/not-done/parse-error arms.

## What's new — `sdk/schema/seam.cue`: two CUE-sourced pod-lifecycle wire carriers (SDD)

The K4 move host-RESOLVES the pod lifecycle plan and the plugin EXECUTES it; two generated wire types
carry that data (regenerated via `task cue:gen` → `spec/cue_types_gen.go`, purely additive):

- **`#PodLifecyclePlan`** — the host-resolved pod start/stop plan the plugin's `OpStart`/`OpStop`
  execute: `mode` (`quadlet`|`direct`), `container_name`, `svc_name`, `run_argv` (the direct-mode
  start argv, host-resolved), `engine_bin`, `unmount`, `enc` (`bytes @go(Enc, type=RawBody)` — the
  opaque enc-op payload the plugin forwards to `verb:enc` via `InvokeProvider`), and
  `tunnel` (`#TunnelConfig @go(Tunnel,optional=nillable)` → `*TunnelConfig`). The arbiter is
  DELIBERATELY absent — exclusive-resource arbitration stays a host-process bracket around the Op (the
  lease lives in host-process state a placement-agnostic plugin cannot own).
- **`#PodExecReply`** — the reply from the plugin's `OpShell` CAPTURED-exec leg (the `charly service`
  move): `{output, exit_code}`. The plugin `RunCapture`s the host-resolved `<engine> exec <ctr>
  <tool> <op> <svc>` argv over the served executor and returns the combined output + the EXACT exit
  code; the host reprints the output (placement-agnostic — an out-of-process plugin's stdout is not
  charly's) and propagates a non-zero `exit_code` as `*ExitCodeError`, preserving the container
  command's exit code exactly through the passthrough→capture semantics change.

Both are plain (or discriminated-by-`mode`) structs `cue exp gengotypes` generates faithfully — no
disjunction, no `@go(-)` hand-write. `RawBody = json.RawMessage` is the established
`bytes @go(…, type=RawBody)` idiom for an opaque forwarded payload.
