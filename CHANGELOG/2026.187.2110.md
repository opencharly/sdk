# CheckContextService.ResolveEndpoint — the generic host-endpoint reverse-leg (H part 2)

Add a class-generic `ResolveEndpoint(port)→addr` RPC to the check-verb reverse channel so an
out-of-process endpoint verb (cdp/vnc/spice/…) resolves its OWN dialable endpoint instead of
the host preresolving it per-verb — the venue/podman/go-libvirt machinery stays host-side, the
plugin decides WHAT to resolve (its port).

- `proto/plugin.proto`: `rpc ResolveEndpoint(ResolveEndpointRequest{port}) returns
  (ResolveEndpointReply{addr,error})` on `CheckContextService` (regenerated stubs). A third
  class-generic RPC — never per-verb (the Uniform API Invariant).
- `kit.CheckContext`: add `ResolveEndpoint(ctx, port) (addr, error)`.
- `checkverb.go`: `sdkCheckContext.ResolveEndpoint` (calls the RPC); new exported
  `NewCheckContext(brokerID, envJSON)` so a RAW `pb.Provider` (Invoke) — not just a
  kit.CheckVerbProvider — can build the reverse-channel CheckContext (single broker Dial).

No CUE-schema change.
