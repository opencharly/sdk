# 2026.190.0755 — deploy/IR machinery lands in the sdk (deploykit + kit + spec + sshx) (P4)

The sdk gains the whole deploy/IR machinery, moved from charly core, so any
plugin importing the sdk can compile deploy plans, construct executors, walk
deploy trees, and read/write the deploy ledger:

- **`deploykit`** — the deploy-plan compiler (`install_build.go` —
  `BuildDeployPlan` + `compile*` helpers, `HostContext`, `MergePlan`,
  `DescribePlan`, `ComputeDeployID`), the InstallStep vocabulary + `step_view.go`,
  the tree walk + executor pick (`deploy_tree.go`/`deploy_chain.go`), the
  deploy-state model (`deploy_state.go`), `plan.go`, `provides.go`,
  `bundle_derive.go`. It reads a candy through the new **`CandyModel` interface**
  (`candy_model.go`) — the read-only view a candy exposes — so the concrete
  runtime `Candy` stays in charly and implements it. `buildkit_aliases.go` pulls
  `ResolvedBox` from `sdk/buildkit` (`deploykit→buildkit` acyclic).
- **`kit`** — executor implementations (`deploy_executor*.go`,
  `deploy_host_shell.go`, `deploy_types.go`), `install_ledger.go`,
  `reverse_ops.go`, `builder_run.go`, the `runtime_config.go` store, and the pure
  helpers (`env`/`envfile`/`http_fetch`/`ports`/`transfer`/`engine`/
  `container_name`/`readiness`).
- **`spec`** — the IR E-envelope (`install_plan.go`, `deploy_executor.go`).
- **`sshx`** — `ssh_client.go` + `ssh_tunnel.go`, re-homed from `vmshared`
  (`vmshared/ssh_target.go` keeps the shared target type).

The charly-coupled compiler legs (provider registry / loader / VerbCatalog) stay
charly-side and are injected as SEAMS (`compile_seams.go`: `CompileActOp`,
`CompileServiceSteps`, `OpInContext`) — deploykit depends on the abstraction, the
concrete kind implements it (kernel/plugin boundary law). Behavior-neutral;
`go test` green; verified against the superproject R10 bed roster.
