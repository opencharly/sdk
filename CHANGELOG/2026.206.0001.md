## What changed

The K1-alpha core-minimization wave relocates `charly`'s shared build-artifact
retention engine (image-tag / build-candy / check-run pruning, plus the
`--deep` store-wide dangling-image purge and the charly-labeled image-tag
inventory) out of `charly/retention.go` and into the compiled-in
`candy/plugin-clean` plugin, where it is served as a new `verb:retention`
capability — the same peer/core-adapter pattern `verb:credential`/`verb:gpu`/
`verb:tunnel` already use, instead of staying core-resident behind a
`HostBuild` seam. This is the sdk-side producer leg of that relocation (the
charly-repo consumer leg is `feat/k1-retention-relocation`, targeting
`feat/k1-unblock-wave1`).

- Extends `#RetentionRequest` (`schema/clean.cue`) with `list` (the read-only
  tag-inventory action backing `charly box list tags`), `build_prune` (the
  narrow post-`charly box build` scope: tag retention + stale
  `.build/_candy` staging dirs only), and `keep_images`/`keep_check_runs`
  (the caller's PRE-RESOLVED `defaults.keep_images`/`keep_check_runs`,
  0 = disabled — the engine no longer reads `charly.yml` itself).
- Adds `#TagInfo` (one locally stored image tag: box/ref/version/in_use) and
  a `tag_groups` field on `#RetentionReply` — the `list` action's payload.
- Both are plain structs — `cue exp gengotypes` generates them faithfully, no
  disjunction/hand-written exception needed (SDD: "wire types are CUE-sourced
  without exception").
- Adds `kit.BuildActivityDir()` — the live-build-activity-lock directory
  primitive, promoted out of charly core (`buildActivityDir`) so charly's
  lock-writer (`acquireBuildActivityLock`) and the relocated retention
  engine's lock-reader (`liveBuildFloor`) share ONE copy instead of a
  core/plugin duplicate (R3).

## Architecture placement (kernel/plugin boundary law)

The retention ENGINE (image-tag/check-run/build-candy pruning, the dangling-
image sweeps, the tag inventory) is a concrete, non-kind-agnostic capability
— an R-item under the boundary law — and correctly belongs in a plugin, not
core; this cutover moves it there. The two wire types this sdk change adds
(`#RetentionRequest`/`#RetentionReply`/`#TagInfo`) are the generic E-envelope
the plugin and its callers exchange across the process/module boundary — kind
-agnostic carriers, never encoding retention-specific business logic in the
kernel. The one piece that legitimately STAYS core-resident (charly-repo
`host_build_retention_defaults.go`, not part of this sdk change) is a minimal
`HostBuild` seam resolving `defaults.keep_images`/`keep_check_runs` via the
core `LoadConfig` loader — the ONE thing the plugin genuinely cannot compute
itself (F11-clean: a class-generic action noun, never a provider word).
