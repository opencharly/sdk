## The two reshapes required for byte-compatible generation

- **`HostProbeReply.GroupAccessible`** (doctor): `map[int]bool` →
  `map[string]bool`. CUE has no int-keyed-map construct (an int-keyed map
  degrades to an empty struct — the documented CAN/CANNOT exception).
  `encoding/json` already renders a `map[int]bool`'s keys as their decimal
  string form on the wire, so `map[string]bool` with the SAME keys is
  byte-identical JSON — a pure representation fix, zero wire-format change.
  Two consumer call sites updated in the companion `opencharly/charly` PR
  (`charly/host_build_hostprobe.go`, `candy/plugin-doctor/command.go`).
- **`VFIOGpu`** (gpu): the former anonymous-embedded `VFIOPCIDevice`
  FLATTENED. CUE unification has no concept of Go embedding — it always
  structurally merges fields — and Go's own JSON-promotion of an anonymous
  embedded struct's fields already made the wire shape identical either way,
  so this is also zero wire-format change. A new `spec.NewVFIOGpu(d)`
  constructor (`spec/gpu_methods.go`) replaces every
  `VFIOGpu{VFIOPCIDevice: d}` composite literal — fixed at all 8 call sites
  across `charly` + `candy/plugin-gpu` in the companion PR.

## RDD catch: `#KeyValue` would have registered a fake "key" kind

The schemagen vocab generator auto-registers EVERY top-level `#<X>Value` CUE
def as a kind-value gate (`kindValueDefs`, `^#?([A-Za-z0-9]+)Value$`). Naming
the deterministic env-var-ordering helper `#KeyValue` would have spuriously
registered a "key" kind — caught live in the regenerated `KindValueDefs` map
before it ever reached a commit. Renamed to `#EnvKV`, with a
`KeyValue = EnvKV` charly-name alias (`spec/charly_names.go`) preserving the
exported Go type name every real consumer (`deploykit.MapToKeyValueSlice`,
`deploykit.SortedEnvList`, `charly/service_render.go`) already uses.
