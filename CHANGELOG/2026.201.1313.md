# 2026.201.1313 — Downloads cache mount follows the task's stage user (curl-23 build failure)

`EmitDownload` always emitted a SHARED root-owned cache mount
(`--mount=type=cache,id=charly-tmp-downloads,dst=/tmp/downloads`) for the
content-addressed downloads cache. The first root-stage download made the
shared backing root-owned, after which EVERY non-root stage's download
failed with `curl: (23) client returned ERROR on write` — a uid-1000 stage
cannot write into a root-owned cache dir. Reproduced serially: any box with
a non-root download step (e.g. the coder images' installer downloads) failed
until the cache was manually purged.

The mount now follows the task's stage user via the new `taskRunsAsRoot`
(root RunAs, or a root image build user when RunAs is empty): non-root
stages get an uid/gid-owned cache (`id=charly-tmp-downloads-uid<uid>` with
`uid=/gid=` — the same per-uid ownership the pixi/npm caches already use),
root stages keep the shared form. `TaskCacheMounts` (candy-declared `cache:`
paths) uses the same decision — it previously treated an empty RunAs as root
(`ResolveUserSpec`'s USER-directive mapping) and shared the same latent bug.

Verified: `TestTaskRunsAsRoot` + `TestEmitDownload_DownloadsCacheOwnership`
(sdk), `TestEmitDownload_DownloadsCacheRootShared` (charly); full standalone
battery green; live proof: a previously curl-23-failing image builds past
the download step with a root-owned shared cache present on the host.

Assisted-by: Kimi Code (Moonshot K3; analysed on a live system)
