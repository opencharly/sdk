# 2026.205.1021 ‚Äî add Deploy.VolumeProjectChecked wire field (companion charly#183 round 3)

`opencharly/charly#183` (the deploy-volume-persistence fix)'s round-2 remediation gated
its project-declared-volume fallback (`resolveDeployVolumes`,
`candy/plugin-deploy-pod`) on per-host overlay-key EXISTENCE ‚Äî but the SAME overlay
entry is also written by unrelated `charly config` stages (the port-resolution block,
`persistResourceCaps`) before the volume fallback ever runs, so "the key exists with an
empty `Volume`" is produced both by "an unrelated write already happened, project never
consulted" (a PORTED deploy's project-declared volume silently dropped on its first
config ‚Äî the round-3 BLOCKING regression, RCA'd by team-lead 2026-07-24) and by "the
project WAS consulted and genuinely declares nothing" (re-querying it on every
subsequent `charly config`/`charly update` ‚Äî the round-2 regression). Overlay-key
existence cannot distinguish the two.

## What changed

Added `volume_project_checked?: bool @go(VolumeProjectChecked)` to `#Deploy`
(`schema/deploy.cue`), immediately following `resolved_port`/`resolved_image` ‚Äî the
SAME "charly-written state, never authored" pattern (a plain persisted marker bit, not
authored by a user, not excluded from `#DeployValue` any more than its two
`resolved_*` siblings already aren't). Regenerated `spec/cue_types_gen.go` via `task
cue:gen` ‚Äî reproducible (a second run produces no diff).

The consumer (`candy/plugin-deploy-pod`'s `resolveDeployVolumes`/`persistDeployVolumes`)
now gates the project fallback on this explicit bit instead of overlay-key presence:
the project is consulted exactly once per deploy key, and its result ‚Äî found or not ‚Äî
is persisted immediately, closing both the round-2 and round-3 regressions at once. See
`opencharly/charly#183`'s own CHANGELOG entry for the full consumer-side fix + R10
evidence.
