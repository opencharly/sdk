## Item 3(b) — `charly box feature run` Kong-nesting: ATTEMPTED, REVERTED

Attempted moving `charly/check_feature_run.go`'s `BoxFeatureCmd`/
`BoxFeatureRunCmd` to `candy/plugin-box` as a 7th `box`-nested command word
(`CommandParent()=="box"`), mirroring plugin-box's existing
generate/validate/new/pkg/inspect/list registration and reaching the
existing `hostFeatureBox` engine over the generic "check-run" HostBuild
seam (zero new host-side surface needed). Implementation compiled cleanly
across all three Go modules (`sdk`, `charly`, `candy/plugin-box`) but
PANICKED AT RUNTIME on process init: `RegisterBuiltinPluginUnit: provider
command:feature already registered`. Root cause (confirmed by reading
`charly/provider_registry.go`): the provider registry's uniqueness key is
`provKey(class, word)` alone, with no `CommandParent` component, and
`candy/plugin-feature` already owns the TOP-LEVEL `{command, feature}` word
(`charly feature list/pending/validate`) — the two "feature" words collide
in the flat registry namespace even though one is nested and the other is
not. REVERTED in full (charly-core's `BoxFeatureCmd`/`BoxFeatureRunCmd`
restored byte-identical to pre-attempt) per the explicit honest-deferral
authorization for this item; routed to P12b with two candidate fixes:
rename the nested word (breaks CLI parity with the current `box feature
run`) or make the registry key `CommandParent`-aware (a cross-cutting core
change touching every nested command, not scoped to this move).
