# 2026.196.0449 — P14a sdk half: #SubstrateStatusRequest/Reply collector wire types + StatusFromState mapper

The sdk half of the P14a status-COLLECTOR move (the consolidated P14-remainder cutover, B2 leg 0).
The cleanly-movable status collectors (pod live + local + the probes) relocate from `charly/` core
into `candy/plugin-substrate` (the super leg), served on the substrate kind provider's `Invoke` by
word. This sdk leg adds the TWO CUE-sourced wire carriers the host↔plugin seam needs, plus the
shared state-normalising mapper — all additive, zero break.

## What's new

- **`schema/status.cue`** — two new self-contained wire defs:
  - **`#SubstrateStatusRequest`** — the per-substrate COLLECTOR request the host sends to the
    substrate plugin's `OpStatusCollect`: the scalar inputs a sdk-only candy cannot derive
    (`engine_bin`, `run_mode`, `quadlet_dir`, `include_all`, and — on the single path —
    `box`+`instance`). NO deploy-cone (`BundleConfig`/`UnifiedFile`) crosses this seam: the deploy
    enrichment stays host-side until K5, applied to the live rows this reply returns.
    vm/k8s/android are deferred to K5 (deploy-cone-coupled); the plugin returns no rows for those
    words until then.
  - **`#SubstrateStatusReply`** — the per-substrate COLLECTOR result: the LIVE rows the substrate
    plugin produced (pod: snapshot-derived + live mounts + tool probes; local: install-ledger
    rows). The host applies the deploy enrichment (tunnel + image-label fallback) to the pod rows
    after they cross this seam; `single` carries the one pod detail row.
- **`spec/cue_types_gen.go`** — regenerated via `task cue:gen` (`cue exp gengotypes`): the two Go
  structs `SubstrateStatusRequest` / `SubstrateStatusReply` (json+yaml tags from `schemagen -mode=retag`).
- **`spec/status_types.go`** — the shared pure mapper **`StatusFromState(state string) string`**:
  normalises an engine/container runtime state vocabulary to charly's `DeploymentStatus.Status`
  vocabulary. Single-sourced (R3) so the substrate plugins' pod-live collector (P14a) AND the
  host-side vm collector (`charly/status_collect_vm.go`, deploy-cone-coupled → stays core until
  K5) read ONE mapper. Pure over its input (no core reentry). This is a NON-wire helper (a pure
  function), NOT a hand-written wire struct — the wire carriers are CUE-generated above.
