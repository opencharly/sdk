# 2026.192.1815 — the build-mode RENDER ENGINE lands in sdk/deploykit (P8 sdk half)

The sdk half of P8 (plugin-build becomes the REAL engine). The build-mode Containerfile
RENDER ENGINE moves from charly core into `sdk/deploykit` as an importable library, so the
render mechanism is no longer core-only — any plugin importing the sdk can render a
Containerfile. It lands in `deploykit` (not `buildkit`) because the render calls the
build-compile helpers (`ResolveCascadePackages`/`CompileLocalPkgStep`/`CompileShellSnippetSteps`)
and reads `deploykit.CandyModel`; the import DAG is one-way `deploykit→buildkit`, so a
buildkit-hosted render would cycle. `buildkit` stays the low keystone layer (`ResolvedBox`,
`RenderTemplate`, `NewInstallContext`).

- **`sdk/deploykit/render_generator.go`** — `Generator`, the build-mode render engine, + the
  host-coupled function-value SEAMS the core resolve half wires (the render reaches every host
  dependency THROUGH a seam so no render-specific per-kind code stays in charly): `EmitPluginOp`
  (Provider-registry verb dispatch), `CollectBoxPorts`/`ValidateEgress` (Config/egress),
  `RenderService` (candy/plugin-init OpResolve), `RewriteHeaderCopyForRemote` +
  `RenderLocalPkgImageInstall` (host-fs), `ResolveInlineBuilder` + `ExternalizedBuilders` (the
  builder-emit registry resolve). `NewRenderGenerator` initializes the per-image reply caches.
- **`deploykit/tasks_render.go` + `tasks_emit.go` + `tasks_stage.go`** — `Generator.EmitTasks`
  + the ~20 per-verb Containerfile-line emitters (`EmitCmd`/`EmitDownload`/`EmitCopy`/
  `EmitVarsEnv`/…) + inline-content staging.
- **`deploykit/candy_steps.go`** — `Generator.WriteCandySteps`, the per-candy render call-graph
  (ENV/packages/localpkg/tasks/inline-builders/shell/user-reset).
- **`deploykit/init.go`** — `Generator.EmitInitAssembly`/`EmitInitFragmentStages`/
  `GenerateInitFragments` + the init template helpers (`InitRender*`, `MapToKeyValueSlice`).
- **`deploykit/candy_stage.go` + `bootstrap.go` + `candy_env.go` + `routes.go`** — scratch/extract
  stages, base bootstrap + dnf-conf, per-candy ENV/EXPOSE, traefik routes.
- **`deploykit/graph.go` + `candy_collect.go` + `effective_version.go` + `order.go` +
  `builder_support.go`** — the build graph (`BoxDirectDeps`), `CollectAllBoxCandies`,
  `ComputeEffectiveVersions`, global-order/base-image resolution, builder detection.
- **`deploykit/candy_model.go`** — `CandyModel` grows the render-delta + init + writeCandySteps
  accessors (`HasInit`/`ServiceFiles`/`RelayPorts`/`HasTasks`/`RunOps`/…); the concrete runtime
  candy stays in charly and implements the interface.
- **`sdk/kit/atomic_write.go` + `calver_compare.go`** — `AtomicWriteFile` + `CompareCalVer`,
  the two pure helpers the render engine + charly share.

Consumer (superproject): charly's render methods become thin delegating shims over
`Generator.toDeploykit()` (the core resolve half wires the seams); the host-side build ENGINE
(Generate/generateContainerfile orchestration, the builder-emit registry cluster, writeLabels
Config-metadata, host-fs/registry seams) stays core as the Mechanism reached via
HostBuild("image"). Behaviour-neutral: `charly box generate` is byte-identical across every
image (proven on the fedora + main image set). Quality gates: `go test ./...` green,
`golangci-lint run` 0 issues, `gofmt` + `go vet` clean.
