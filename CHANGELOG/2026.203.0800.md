## Fixed

- **R3 — hoisted the duplicated `loadBundleConfig`-style seam read into ONE shared
  `deploykit.LoadBundleConfigViaSeam`.** The bed-robustness batch's first charly leg
  (`charly#176`) landed two brand-new copies of the identical "marshal
  `spec.PodConfigLoadDeployRequest` → `ex.HostBuild(ctx, "pod-config-load-bundle", …)` →
  unmarshal `spec.PodConfigLoadBundleReply` → unmarshal `ConfigJSON` into a
  `deploykit.BundleConfig`" sequence (`candy/plugin-status/nested_tree.go`,
  `candy/plugin-substrate/status_flat.go`) alongside two PRE-EXISTING copies of the same
  pattern (`candy/plugin-bundle/ephemeral.go`, `candy/plugin-pod/remove_orchestration.go`)
  — a fresh `pr-validator` review of `charly#176` (round 1) correctly rejected landing a
  3rd and 4th copy of one pattern in a single cutover as R3's exact trigger, and rejected
  the "plugin modules can't cross-import each other" justification: every one of the four
  call sites already imports `sdk/deploykit` + `sdk/spec` + the root `sdk` package freely,
  and an sdk kit is exactly the mechanism this project uses to share code across plugin
  module boundaries.
- New file `deploykit/load_bundle_config_seam.go` — `LoadBundleConfigViaSeam(ctx, ex,
  caller)` is the ONE shared implementation. It takes the caller's already-resolved
  reverse-channel executor (a package-var stash for a single-command-per-process command
  plugin, or a per-call `sdk.ExecutorForInvoke` result for a multi-call verb provider) and
  a short caller label for host-side diagnostics — HOW a caller obtains its executor stays
  its own placement concern, not this seam's. `deploykit` already imports the root
  `github.com/opencharly/sdk` package elsewhere (`render_generator_from_project.go`, for
  the identical sdk.Executor/HostBuild reason), so this adds no new import cycle.

## Behavior

**Pure consolidation — byte-equivalent semantics.** Every call site's observable behavior
is unchanged: same HostBuild seam kind (`pod-config-load-bundle`), same request/reply
wire shape, same nil-on-empty-overlay contract, same error propagation on a
HostBuild/decode failure. Only the caller-facing error TEXT is now emitted from one place
instead of four near-identical copies — no test in this repo or the `charly` consumer
repo asserts on that text (the seam-coupled load itself has never been unit-testable
without a live reverse channel at any of the four call sites; each site instead tests only
the pure extraction/merge logic downstream of it). This PR ships its own direct coverage
of the new shared function (`load_bundle_config_seam_test.go`, 4 cases: non-empty
round-trip, nil-on-empty, HostBuild transport-error propagation, nil-executor guard) using
the same `sdk.NewInProcExecutor` + fake `pb.ExecutorServiceClient` wiring
`candy/plugin-deploy-vm/lifecycle_test.go` already established for this class of seam
test.

No behavior change means the `charly` consumer leg's repoint of its four call sites onto
this function needs only a delta re-gate (rebuild + `go test` on the touched modules) —
not a disposable-bed re-run — per the operator's explicit ruling on this round.
